## Секция 1: TraceId & SpanId в логах

### Задача 9.1: Включить автоматическую инъекцию TraceId/SpanId в логи через OTel

**Описание:** Изучить интеграцию логирования и трассировки в OTel SDK. Понять, что при активном трейсе `trace_id` и `span_id` автоматически добавляются в логи. Убедиться, что `OpenTelemetry.Extensions.Hosting` подключен для этой функциональности.

**Способы достижения:**

- Проверить наличие пакета: `dotnet list package`
- Запустить код с активным трейсом: `using var activity = Source.StartActivity("Test")`
- Вывести лог внутри трейса и проверить JSON: поля `trace_id` и `span_id` должны присутствовать
- Закоммитить: `git add .`
  `git commit -m "feat(module9): enable TraceId/SpanId injection in logs"`

**Результат:** JSON-лог с полями `trace_id` и `span_id`, коммит с включением автоматической инъекции в репозитории.

📚 **Теоретическая справка**

- **Что это?** Механизм автоматического обогащения записей лога идентификаторами распределённой трассировки из текущего контекста `Activity`.
- **Зачем это?** Обеспечивает связность между логами и трейсами: по `trace_id` можно найти все логи, относящиеся к одному запросу, что критично для отладки распределённых систем.
- **Как это реализовано?** `OpenTelemetry.Extensions.Hosting` регистрирует `LogRecordEnricher`, который при каждом вызове `ILogger.Log` проверяет `Activity.Current`. Если активность существует, обогащает `LogRecord` полями `trace_id`, `span_id`, `trace_flags`. При использовании `JsonConsoleFormatter` эти поля сериализуются как часть JSON-объекта.
- **Экосистема и альтернативы**

| Механизм связки логов и трейсов              | Автоматизация                       | Область применения                          |
| -------------------------------------------- | ----------------------------------- | ------------------------------------------- |
| `LogRecordEnricher` (OTel)                   | Полная (прозрачная для кода)        | ✅ Стандартный подход, минимальный оверхед  |
| Ручное добавление `Activity.Current.TraceId` | Низкая (требует кода в каждом логе) | ❌ Многословно, подвержен ошибкам           |
| `Serilog.Enrichers.WithSpan`                 | Полная (через стороннюю библиотеку) | ❌ Привязка к Serilog, не стандарт          |
| CorrelationId через `HttpContext`            | Частичная (только для веб)          | ❌ Не работает в консольных/фоновых задачах |

## Секция 2: CorrelationId — бизнес-идентификатор запроса

### Задача 9.2: Реализовать извлечение и логирование CorrelationId из Activity

**Описание:** Понять разницу между техническим `TraceId` и бизнес-`CorrelationId`. Реализовать извлечение `TraceId` из `Activity.Current` как fallback для CorrelationId. Логировать CorrelationId как отдельное поле для удобства фильтрации.

**Способы достижения:**

- Извлечь: `var correlationId = Activity.Current?.TraceId.ToString() ?? Guid.NewGuid().ToString()`
- Логировать: `logger.LogInformation("Processing with CorrelationId={CorrelationId}", correlationId)`
- Проверить совпадение: поле `CorrelationId` в логе должно соответствовать `trace_id`
- Закоммитить: `git commit -am "feat(module9): implement CorrelationId logging from Activity"`

**Результат:** Лог с полем `CorrelationId`, совпадающим с `trace_id`, коммит с реализацией бизнес-идентификатора.

📚 **Теоретическая справка**

- **Что это?** Бизнес-ориентированный идентификатор, связывающий все события, относящиеся к одной пользовательской операции или запросу.
- **Зачем это?** Упрощает поиск и фильтрацию логов в агрегаторах: оператор может найти все события по одному `CorrelationId` без знания внутренней структуры трейсов.
- **Как это реализовано?** `Activity.TraceId` — 128-битный идентификатор, представленный как `ActivityTraceId` struct. Метод `ToString()` возвращает шестнадцатеричное строковое представление. При отсутствии активной активности генерируется новый `Guid` как запасной вариант. Поле логируется через message template, что сохраняет его как отдельный ключ в структурированном выводе.
- **Экосистема и альтернативы**

| Стратегия идентификации запроса       | Уникальность                         | Область применения                                       |
| ------------------------------------- | ------------------------------------ | -------------------------------------------------------- |
| `Activity.TraceId` (128-bit)          | Глобальная (вероятность коллизии ~0) | ✅ Распределённая трассировка, стандарт OTel             |
| `Guid.NewGuid()` (128-bit)            | Глобальная                           | ✅ Локальные корреляции, не связаны с трейсами           |
| `short GUID` / Base64                 | Частичная (риск коллизий)            | ❌ Короткие идентификаторы для логов, не для трассировки |
| Пользовательский ID (email, order_id) | Бизнес-уникальность                  | ❌ Не подходит как технический ключ, может повторяться   |

## Секция 3: Baggage — кастомный контекст в трейсах

### Задача 9.3: Настроить передачу кастомных данных через Baggage API

**Описание:** Изучить OTel Baggage API для передачи произвольных пар key-value через контекст. Понять, что Baggage доступен асинхронно через `AsyncLocal`. Реализовать установку и чтение значений багажа в коде.

**Способы достижения:**

- Установить: `Baggage.SetBaggage("user.id", "123").SetBaggage("tenant.id", "abc")`
- Прочитать: `var userId = Baggage.Current.GetBaggage("user.id")`
- Проверить в отладчике или через лог, что значения доступны после асинхронного перехода
- Закоммитить: `git commit -am "feat(module9): use Baggage for custom context propagation"`

**Результат:** Код установки/чтения Baggage, подтверждение доступности значений после async-перехода, коммит с реализацией.

📚 **Теоретическая справка**

- **Что это?** Механизм хранения произвольных метаданных в контексте выполнения, автоматически пробрасываемый через асинхронные границы и сетевые вызовы.
- **Зачем это?** Позволяет передавать бизнес-контекст (user_id, tenant_id, feature_flag) через слои приложения без явной передачи параметров в каждый метод.
- **Как это реализовано?** `Baggage` использует `AsyncLocal<BaggageWrapper>` для хранения словаря ключ-значение в контексте `ExecutionContext`. При асинхронном переходе (`await`) `ExecutionContext` копируется, сохраняя доступность багажа. Методы `SetBaggage` возвращают новый экземпляр `Baggage` (immutable), который становится текущим через `Baggage.Current = ...`. При экспорте трейса baggage-пары сериализуются как атрибуты спана.
- **Экосистема и альтернативы**

| Механизм передачи контекста | Область видимости                                | Сериализация в сеть                             |
| --------------------------- | ------------------------------------------------ | ----------------------------------------------- |
| `Baggage` (OTel)            | AsyncLocal + автоматическая инъекция в заголовки | ✅ W3C Baggage header, совместим с провайдерами |
| `AsyncLocal<T>` (ручной)    | Только процесс, нет авто-сериализации            | ❌ Требует ручной передачи в заголовки          |
| Явные параметры метода      | Локальный стек вызовов                           | ❌ Многословно, нарушает инкапсуляцию           |
| `HttpContext.Items`         | Только HTTP-запрос                               | ❌ Не работает вне веб-контекста                |

## Секция 4: Сквозная проверка корреляции контекста

### Задача 9.4: Провести интеграционный тест сквозного проброса CorrelationId и Baggage

**Описание:** Собрать сценарий: запуск трейса → установка Baggage → асинхронный вызов → логирование в целевом методе. Убедиться, что CorrelationId и Baggage доступны в асинхронном методе без явной передачи параметров.

**Способы достижения:**

- Создать метод `async Task ProcessAsync()` с логированием `CorrelationId` и значений из Baggage
- Вызвать метод внутри активного трейса с установленным багажом
- Проверить логи: `CorrelationId` и `user.id` должны совпадать с установленными значениями
- Закоммитить: `git add .`
  `git commit -m "feat(module9): verify end-to-end correlation propagation"`

**Результат:** Логи из разных методов с одинаковым `CorrelationId` и данными из Baggage, коммит с интеграционным тестом.

📚 **Теоретическая справка**

- **Что это?** Валидация целостности контекста выполнения при переходе через асинхронные границы, гарантирующая, что метаданные не теряются в цепочке вызовов.
- **Зачем это?** Подтверждает корректность архитектуры наблюдаемости: если контекст теряется, отладка распределённых сценариев становится невозможной, а логи — несопоставимыми.
- **Как это реализовано?** Тест создаёт `Activity`, устанавливает `Baggage`, затем вызывает асинхронный метод. Благодаря `ExecutionContext.Flow`, `AsyncLocal`-хранилища копируются в продолжение задачи. Внутри `ProcessAsync` обращение к `Activity.Current` и `Baggage.Current` возвращает исходные значения. Логи, обогащённые этими данными, подтверждают сквозную связность.
- **Экосистема и альтернативы**

| Метод валидации контекста                | Покрытие                   | Сложность                                             |
| ---------------------------------------- | -------------------------- | ----------------------------------------------------- |
| Интеграционный тест с `InMemoryExporter` | Полный (включая экспорт)   | ✅ Низкая, встроена в тестовый проект                 |
| `Assert` на `AsyncLocal` значениях       | Частичный (только процесс) | ❌ Не проверяет сериализацию в сеть                   |
| E2E-тест с OTel Collector                | Полный + инфраструктура    | ❌ Требует развёрнутого бэкенда, избыточно для модуля |
| Статический анализ кода                  | Нулевой (только синтаксис) | ❌ Не проверяет runtime-поведение                     |

## Секция 5: Propagators — сериализация контекста для сети

### Задача 9.5: Исследовать механизмы пропагации контекста через границы процессов

**Описание:** Изучить стандарт W3C TraceContext и механизм Propagators для сериализации контекста в заголовки. Понять разницу между инъекцией и извлечением контекста. Реализовать ручной пример для консольного приложения.

**Способы достижения:**

- Использовать: `Propagators.DefaultTextMapPropagator.Inject(Activity.Current, carrier, Setter)`
- Симулировать извлечение: создать карьер с заголовками, вызвать `Extract`
- Проверить, что восстановленный `Activity` содержит исходный `TraceId` и Baggage
- Закоммитить: `git commit -am "chore(module9): explore manual context propagation via Propagators"`

**Результат:** Код ручной инъекции/извлечения контекста, подтверждение восстановления `TraceId`, коммит с исследованием пропагаторов.

📚 **Теоретическая справка**

- **Что это?** Компоненты, отвечающие за сериализацию контекста трассировки в текстовые заголовки (для HTTP/gRPC) и обратное восстановление из входящих запросов.
- **Зачем это?** Обеспечивает сквозную трассировку между микросервисами, приложениями и внешними системами, сохраняя связность трейсов через сетевые границы.
- **Как это реализовано?** `TextMapPropagator` определяет интерфейс для работы с произвольным "карьером" (словарь заголовков). Метод `Inject` извлекает `traceparent` и `tracestate` из `Activity`, форматирует их по спецификации W3C и записывает в карьер. `Extract` парсит входящие заголовки, создаёт новую `Activity` с восстановленным `TraceId` и `ParentId`, привязывая её к текущему контексту.
- **Экосистема и альтернативы**

| Стандарт пропагации              | Формат заголовков                 | Совместимость                                           |
| -------------------------------- | --------------------------------- | ------------------------------------------------------- |
| W3C TraceContext (`traceparent`) | `00-{trace-id}-{span-id}-{flags}` | ✅ Кросс-вендорный стандарт, рекомендован OTel          |
| B3 (Zipkin)                      | `X-B3-TraceId`, `X-B3-SpanId`     | ❌ Устаревший, требует миграции                         |
| Jaeger                           | `uber-trace-id`                   | ❌ Привязка к Jaeger, не стандарт                       |
| Custom-заголовки                 | Произвольный формат               | ❌ Не совместим с экосистемой, требует ручной поддержки |
