## Секция 1: Инициализация OTel SDK

### Задача 8.1: Настроить базовую инициализацию OpenTelemetry SDK с провайдерами сигналов

**Описание:** Изучить архитектуру OTel SDK: `TracerProvider`, `MeterProvider`, `LoggerProvider`. Подключить необходимые NuGet-пакеты и инициализировать SDK через `AddOpenTelemetry()`. Настроить минимальную конфигурацию для трейсов и метрик.

**Способы достижения:**

- Добавить пакеты: `dotnet add package OpenTelemetry.Exporter.Console OpenTelemetry.Exporter.OpenTelemetryProtocol`
- В `Program.cs` добавить: `builder.Services.AddOpenTelemetry().WithTracing(t => t.AddSource("EP.ConsoleStarter")).WithMetrics(m => m.AddMeter("EP.ConsoleStarter"))`
- Проверить сборку: `dotnet build` без ошибок, убедиться в подключении пакетов
- Закоммитить: `git add .`
  `git commit -m "feat(module8): add OpenTelemetry SDK initialization"`

**Результат:** Блок `AddOpenTelemetry` в Program.cs, успешная сборка, коммит с инициализацией SDK в репозитории.

📚 **Теоретическая справка**

- **Что это?** Единая точка входа для настройки инфраструктуры наблюдаемости, регистрирующая провайдеры трейсов, метрик и логов в контейнере зависимостей.
- **Зачем это?** Стандартизирует инструментацию приложений, обеспечивает совместимость с бэкендами (Jaeger, Prometheus, OTLP), упрощает замену экспортеров без изменения бизнес-логики.
- **Как это реализовано?** `AddOpenTelemetry` расширяет `IServiceCollection`, регистрируя синглтоны `TracerProvider`, `MeterProvider`, `LoggerProvider`. Методы `WithTracing`/`WithMetrics` конфигурируют источники данных (`ActivitySource`, `Meter`) и цепочку процессоров/экспортеров. При старте хоста провайдеры инициализируются и начинают приём сигналов.
- **Экосистема и альтернативы**

| Компонент инициализации                     | Сигналы               | Область применения                                |
| ------------------------------------------- | --------------------- | ------------------------------------------------- |
| `AddOpenTelemetry()` (OTel SDK)             | Traces, Metrics, Logs | ✅ Кросс-языковой стандарт, универсальный экспорт |
| `AddApplicationInsights()`                  | Traces, Metrics, Logs | ❌ Привязка к Azure, проприетарный бэкенд         |
| Ручная регистрация `ActivitySource`/`Meter` | Traces, Metrics       | ❌ Требует ручной настройки экспортеров           |
| `Serilog` + обогащения                      | Только логи           | ❌ Не покрывает трейсы и метрики                  |

## Секция 2: ActivitySource для трассировки

### Задача 8.2: Реализовать создание трейсов через ActivitySource с тегами

**Описание:** Изучить класс `ActivitySource` как фабрику для создания трейсов. Реализовать статический экземпляр с именем приложения. Научиться начинать спаны через `StartActivity()` и добавлять теги через `SetTag()`.

**Способы достижения:**

- Объявить: `private static readonly ActivitySource Source = new("EP.ConsoleStarter", "1.0.0")`
- В бизнес-логике: `using var activity = Source.StartActivity("ProcessData"); activity?.SetTag("item.count", 10)`
- Добавить `AddConsoleExporter()` в конфигурацию трейсов и увидеть вывод спана в консоли
- Закоммитить: `git commit -am "feat(module8): implement ActivitySource tracing with tags"`

**Результат:** Консольный вывод трейса с тегом `item.count=10`, коммит с реализацией трассировки в репозитории.

📚 **Теоретическая справка**

- **Что это?** Фабрика для создания объектов `Activity`, представляющих единицы работы (спаны) в распределённой трассировке.
- **Зачем это?** Позволяет инструментировать бизнес-операции, измерять их длительность, фиксировать метаданные и связывать с родительскими/дочерними операциями для построения графа выполнения.
- **Как это реализовано?** `ActivitySource` регистрируется в `TracerProvider` по имени. Вызов `StartActivity` создает экземпляр `Activity` с уникальным `SpanId`, привязывает его к `AsyncLocal` для автоматического наследования контекста. Метод `SetTag` добавляет ключ-значение в коллекцию тегов, которые сериализуются при экспорте. `using`-блок гарантирует завершение спана и фиксацию длительности.
- **Экосистема и альтернативы**

| Механизм трассировки                      | Уровень абстракции        | Область применения                                             |
| ----------------------------------------- | ------------------------- | -------------------------------------------------------------- |
| `ActivitySource` (OTel)                   | Высокий, стандартный      | ✅ Распределённая трассировка, кросс-платформенная             |
| `DiagnosticSource`                        | Низкий, событийный        | ❌ Внутренние события .NET, не для бизнес-логики               |
| `System.Diagnostics.Trace`                | Legacy, процесс-локальный | ❌ Устаревший, не поддерживает распределённый контекст         |
| Сторонние APM-агенты (Datadog, New Relic) | Очень высокий             | ❌ Авто-инструментация, но требует агента и привязки к вендору |

## Секция 3: Meter для кастомных метрик

### Задача 8.3: Создать инструменты метрик через Meter API

**Описание:** Понять, что `Meter` — фабрика для инструментов метрик: Counter, Histogram, Gauge. Создать статический экземпляр Meter с именем приложения. Реализовать счётчик `items.processed` и инкрементировать его в коде.

**Способы достижения:**

- Объявить: `private static readonly Meter Meter = new("EP.ConsoleStarter", "1.0.0")`
- Создать инструмент: `private static readonly Counter<long> ItemsProcessed = Meter.CreateCounter<long>("items.processed")`
- Инкрементировать: `ItemsProcessed.Add(1, new("item.type", "report"))`
- Добавить `AddConsoleExporter()` для метрик и увидеть пуш в консоли
- Закоммитить: `git commit -am "feat(module8): implement Meter for custom metrics"`

**Результат:** Консольный вывод метрики `items.processed` с атрибутом `item.type=report`, коммит с настройкой метрик.

📚 **Теоретическая справка**

- **Что это?** Фабрика для создания инструментов сбора метрик: счётчиков, гистограмм, измерителей текущего состояния.
- **Зачем это?** Позволяет измерять бизнес-показатели (количество обработанных элементов, ошибки, задержки) с детализацией по атрибутам для последующей агрегации и алертинга.
- **Как это реализовано?** `Meter` регистрируется в `MeterProvider` по имени. Инструменты (`Counter<T>`, `Histogram<T>`, `Gauge<T>`) создаются через фабричные методы и хранят ссылку на провайдер. Вызов `Add`/`Record`/`Set` немедленно обновляет внутреннее состояние и уведомляет экспортеры. Атрибуты передаются как `ReadOnlySpan<KeyValuePair<string, object>>` для минимизации аллокаций.
- **Экосистема и альтернативы**

| Тип метрики        | Поведение                                    | Область применения                                 |
| ------------------ | -------------------------------------------- | -------------------------------------------------- |
| `Counter<T>`       | Монотонный рост (только инкремент)           | ✅ События: запросы, ошибки, обработки             |
| `Histogram<T>`     | Распределение значений (min/max/sum/buckets) | ✅ Длительность операций, размеры данных           |
| `Gauge<T>`         | Произвольное значение (рост/падение)         | ✅ Текущее состояние: очередь, память, подключения |
| `UpDownCounter<T>` | Счётчик с инкрементом/декрементом            | ❌ Редкие сценарии: баланс, дельты                 |

## Секция 4: Интеграция экспортеров Console и OTLP

### Задача 8.4: Настроить вывод сигналов в консоль и через OTLP-протокол

**Описание:** Собрать полную конфигурацию экспортеров для трёх сигналов: Traces, Metrics, Logs. Настроить `AddConsoleExporter` для отладки и `AddOtlpExporter` для продакшена. Убедиться, что данные корректно выводятся в обоих случаях.

**Способы достижения:**

- В конфигурации: `.WithTracing(t => t.AddConsoleExporter().AddOtlpExporter())` и аналогично для Metrics/Logs
- Запустить `dotnet run` и увидеть вывод трейсов, метрик и логов в консоли
- Проверить конфигурацию OTLP: `options.Endpoint = new Uri("http://localhost:4317")`
- Закоммитить: `git add .`
  `git commit -m "feat(module8): configure Console & OTLP exporters"`

**Результат:** Вывод Console Exporter с тремя сигналами, конфигурация OTLP endpoint, коммит с интеграцией экспортеров.

📚 **Теоретическая справка**

- **Что это?** Компоненты, отвечающие за сериализацию и передачу сигналов наблюдаемости во внешние системы: консоль для отладки, OTLP для агрегации.
- **Зачем это?** Обеспечивает гибкость: локальная разработка с визуальной проверкой, production-экспорт в централизованные бэкенды без изменения кода инструментации.
- **Как это реализовано?** Экспортеры регистрируются в провайдерах как конечные обработчики. `ConsoleExporter` форматирует данные в человекочитаемый текст и пишет в `Console.Out`. `OtlpExporter` сериализует сигналы в Protobuf, отправляет через gRPC/HTTP на указанный endpoint. Оба экспортера могут работать параллельно: данные клонируются на этапе процессинга.
- **Экосистема и альтернативы**

| Экспортер                           | Протокол              | Область применения                         |
| ----------------------------------- | --------------------- | ------------------------------------------ |
| `ConsoleExporter`                   | Текст / JSON в stdout | ✅ Отладка, CI/CD, локальная разработка    |
| `OtlpExporter`                      | gRPC / HTTP+Protobuf  | ✅ Production, интеграция с OTel Collector |
| `PrometheusExporter`                | HTTP-скрейпинг        | ❌ Только метрики, pull-модель             |
| `ZipkinExporter` / `JaegerExporter` | Специфичные протоколы | ❌ Устаревшие, требуют миграции на OTLP    |

## Секция 5: Настройка OTLP-протокола и endpoint

### Задача 8.5: Явно указать протокол и endpoint для OTLP-экспорта

**Описание:** Изучить различия между gRPC и HTTP/Protobuf в OTLP. Понять стандартный endpoint `localhost:4317` для gRPC. Настроить явное указание протокола в конфигурации экспортера для предсказуемости поведения.

**Способы достижения:**

- В конфигурации: `AddOtlpExporter(opt => { opt.Protocol = OtlpExportProtocol.Grpc; opt.Endpoint = new Uri("http://localhost:4317"); })`
- Проверить, что `OtlpExportProtocol` указан явно, не по умолчанию
- При наличии: запустить локальный OTel Collector и проверить приём данных
- Закоммитить: `git commit -am "chore(module8): configure OTLP gRPC endpoint explicitly"`

**Результат:** Конфиг `AddOtlpExporter` с явным `OtlpExportProtocol.Grpc` и endpoint, коммит с настройкой протокола.

📚 **Теоретическая справка**

- **Что это?** Конфигурация транспортного уровня для экспорта данных наблюдаемости через стандартный протокол OpenTelemetry Protocol (OTLP).
- **Зачем это?** Гарантирует совместимость с целевым бэкендом, исключает неявные зависимости от дефолтных значений, упрощает миграцию между окружениями (dev → prod).
- **Как это реализовано?** `OtlpExportProtocol` определяет транспорт: `Grpc` (порт 4317) или `HttpProtobuf` (порт 4318). При инициализации экспортер создает `HttpClient` или `GrpcChannel` с указанным endpoint, настраивает таймауты, заголовки авторизации. Данные сериализуются в Protobuf и отправляются в потоковом или пакетном режиме.
- **Экосистема и альтернативы**

| Транспорт OTLP                             | Порт    | Преимущества                                                   |
| ------------------------------------------ | ------- | -------------------------------------------------------------- |
| `gRPC` (4317)                              | 4317    | ✅ Эффективный бинарный протокол, стриминг, обратно совместим  |
| `HTTP/Protobuf` (4318)                     | 4318    | ✅ Проще для прокси/фаерволов, совместим с веб-инфраструктурой |
| `JSON over HTTP`                           | Любое   | ❌ Устаревший, не рекомендуется спецификацией                  |
| Проприетарные протоколы (InfluxDB, StatsD) | Зависит | ❌ Привязка к вендору, не стандарт                             |
