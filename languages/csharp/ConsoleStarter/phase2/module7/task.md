## Секция 1: ILogger<T> и уровни логирования

### Задача 7.1: Настроить инжект логгера с корректными категориями и уровнями

**Описание:** Изучить систему логирования .NET и инжект `ILogger<T>` через встроенный DI. Понять, что категория лога автоматически соответствует полному имени типа. Реализовать вызовы логов разных уровней для покрытия всех сценариев.

**Способы достижения:**

- Инжектить `ILogger<Program>` в точку входа через параметры метода или резолв из `host.Services`
- Добавить вызовы: `_logger.LogTrace()`, `_logger.LogInformation()`, `_logger.LogWarning()`, `_logger.LogError()`
- Запустить приложение и проверить вывод: категории вида `EP.ConsoleStarter.Program`, уровни отображаются корректно
- Закоммитить: `git add . && git commit -m "feat(module7): implement ILogger<T> injection with levels"`

**Результат:** Консольный вывод с логами всех уровней и корректными категориями, коммит с настройкой логгера в истории репозитория.

## Секция 2: Message Templates — структурированные параметры

### Задача 7.2: Реализовать логирование через message templates вместо строковой интерполяции

**Описание:** Понять разницу между `ILogger` template `{UserId}` и строковой интерполяцией `$"..."`. Научиться передавать параметры как именованные аргументы для сохранения структуры. Убедиться, что при JSON-форматировании параметры становятся отдельными полями.

**Способы достижения:**

- Заменить интерполяцию: `logger.LogInformation($"User {id} logged")` → `logger.LogInformation("User {UserId} logged", id)`
- Добавить несколько параметров: `logger.LogInformation("User {UserId} from {Ip} accessed {Resource}", id, ip, resource)`
- Настроить временный JSON-форматтер и проверить вывод: поля `UserId`, `Ip`, `Resource` должны быть отдельными ключами
- Закоммитить: `git commit -am "feat(module7): use structured message templates"`

**Результат:** JSON-лог с отдельными полями `"UserId": 123`, `"Ip": "10.0.0.1"`, коммит с переходом на template-логирование.

## Секция 3: Logging Scopes — контекстная группировка

### Задача 7.3: Реализовать логические группы событий через BeginScope

**Описание:** Изучить механизм `ILogger.BeginScope()` для добавления контекста ко всем логам внутри блока. Понять, что scope идеален для отслеживания операций/транзакций. Реализовать scope с уникальным идентификатором для группировки связанных событий.

**Способы достижения:**

- Обернуть блок кода: `using (logger.BeginScope(new { TransactionId = Guid.NewGuid() })) { ... }`
- Внутри scope добавить несколько логов и убедиться, что `TransactionId` присутствует в каждом
- Проверить вложенные scope: контекст должен накапливаться, не перезаписываться
- Закоммитить: `git commit -am "feat(module7): implement logging scopes for operation tracking"`

**Результат:** Два и более JSON-лога с одинаковым `TransactionId` внутри scope, коммит с реализацией контекстной группировки.

## Секция 4: Интеграция JsonConsoleFormatter

### Задача 7.4: Настроить глобальный JSON-форматтер для консольного вывода

**Описание:** Заменить текстовый вывод логгера на структурированный JSON через `JsonConsoleFormatter`. Настроить опции форматирования: отступы, включение скоупов, формат временных меток. Убедиться, что все логи приложения, включая системные, выводятся в валидном JSON.

**Способы достижения:**

- В `Program.cs` настроить: `builder.Logging.ClearProviders().AddJsonConsole(opt => opt.JsonWriterOptions.Indented = true)`
- Запустить `dotnet run | jq .` и проверить, что каждый выводимый лог — валидный JSON-объект
- Убедиться, что поля `Timestamp`, `LogLevel`, `Category`, `Message`, параметры шаблона — все присутствуют
- Закоммитить: `git add . && git commit -m "feat(module7): configure JsonConsoleFormatter globally"`

**Результат:** Валидный JSON-вывод всего приложения, прошедший проверку `jq`, коммит с настройкой форматтера в репозитории.

## Секция 5: Диагностика через dotnet-trace

### Задача 7.5: Настроить сбор диагностических событий без изменения кода

**Описание:** Изучить инструмент `dotnet-trace` для сбора событий логирования в рантайме. Понять провайдер `Microsoft-Extensions-Logging` и формат `nettrace`. Научиться конвертировать трассировку для визуального анализа в Speedscope.

**Способы достижения:**

- Запустить приложение в фоне: `dotnet run &`
- Собрать трассировку: `dotnet trace collect --process-id <pid> --providers Microsoft-Extensions-Logging`
- Конвертировать и открыть в `speedscope`: `dotnet trace convert trace.nettrace`
- Закоммитить артефакты: `git add diagnostics/ && git commit -m "chore(module7): add diagnostic tooling setup"`

**Результат:** Файл `trace.nettrace` и скриншот из Speedscope с таймлайном лог-событий, коммит с настройкой диагностики.
