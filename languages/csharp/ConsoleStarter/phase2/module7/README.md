# Модуль 7 — 📝 Structured Logging Basics

> Цель: Настроить production-ready наблюдаемость через `JsonConsoleFormatter`, message templates и logging scopes.

---

## 🚀 Быстрый старт

### Сборка и запуск (разработка)

```powershell
# Из корня репозитория
cd src/ConsoleStarter
dotnet run
```

**Ожидаемый вывод:**
Каждая строка должна быть валидным JSON-объектом (начинаться с `{` и заканчиваться `}`).

Пример корректного структурного лога:

```json
{
  "EventId": 0,
  "LogLevel": "Information",
  "Category": "Program",
  "Message": "User {UserId} from {Ip} accessed {Resource}",
  "State": {
    "UserId": 123,
    "Ip": "10.0.0.1",
    "Resource": "/api/data",
    "{OriginalFormat}": "User {UserId} from {Ip} accessed {Resource}"
  },
  "Scopes": ["{ TransactionId = a1b2c3d4... }"]
}
```

### Остановка

Нажми `Ctrl+C` → приложение корректно завершится, хост запишет логи остановки в том же JSON-формате.

## ✅ Критерии приёмки

- [x] `ILogger<T>` настроен через встроенный DI, категории формируются автоматически из типа `T`.
- [x] Вывод `dotnet run` состоит из валидных JSON-строк (текстовые `Console.WriteLine` удалены или минимизированы).
- [x] Параметры логов передаются через message templates (`{Param}`) и сохраняются как отдельные типизированные поля в `State`, а не склеиваются в строку `Message`.
- [x] `BeginScope` с опцией `IncludeScopes = true` корректно добавляет контекст операции в массив `Scopes` без явной передачи параметров в методы.
- [x] Конфигурация `JsonConsoleFormatter` задана глобально (`Indented = false` для продакшена).
- [x] Документация (`README.md`) и трекер (`PROGRESS.md`) заполнены, PR создан.

## 🔑 Ключевые инсайты

| Концепция            | Инсайт                                                                                                                                                                             |
| -------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Message Template** | `{UserId}` вместо `$"{UserId}"`. Сохраняет типизацию (`int`, `string`) для индексации агрегаторами (ELK, Seq, OTel). Интерполяция убивает структуру.                               |
| **Logging Scope**    | `using var scope = logger.BeginScope(...)` гарантирует привязку контекста (напр., `TransactionId`) ко всем логам блока. Не требует засорения сигнатур методов лишними параметрами. |
| **JSON Formatter**   | `IncludeScopes = true` критически важен, иначе контекст теряется. `Indented = false` обязателен в продакшене (экономит байты, ускоряет парсинг).                                   |
| **Диагностика**      | `dotnet-trace` позволяет собирать логи и события без изменения кода и перезапуска процесса, что идеально для отладки продакшен-инцидентов.                                         |

## 🧪 Проверка вручную

1. **Штатный запуск и валидация JSON:**

   ```powershell
   cd src/ConsoleStarter
   dotnet run
   # Визуально убедись, что весь вывод — это однострочные JSON-объекты.
   # (Опционально, если установлен jq): dotnet run | jq . > $null
   ```

2. **Проверка структуры:**
   Найди в выводе лог с параметрами. Убедись, что в объекте есть:
   - Поле `"Message"` с исходным шаблоном (или `{OriginalFormat}` в `State`).
   - Отдельные ключи в `"State"` (например, `"UserId": 123`).
   - Массив `"Scopes"` с активным контекстом (например, `TransactionId`).
