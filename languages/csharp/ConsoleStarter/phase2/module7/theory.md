## Секция 1: `ILogger<T>` и уровни логирования

### Задача 7.1: Настроить инжект логгера с корректными категориями и уровнями

**Описание:** Изучить систему логирования .NET и инжект `ILogger<T>` через встроенный DI. Понять, что категория лога автоматически соответствует полному имени типа. Реализовать вызовы логов разных уровней для покрытия всех сценариев.

**Способы достижения:**

- Инжектить `ILogger<Program>` в точку входа через параметры метода или резолв из `host.Services`
- Добавить вызовы: `_logger.LogTrace()`, `_logger.LogInformation()`, `_logger.LogWarning()`, `_logger.LogError()`
- Запустить приложение и проверить вывод: категории вида `EP.ConsoleStarter.Program`, уровни отображаются корректно
- Закоммитить: `git add .`
  `git commit -m "feat(module7): implement ILogger<T> injection with levels"`

**Результат:** Консольный вывод с логами всех уровней и корректными категориями, коммит с настройкой логгера в истории репозитория.

📚 **Теоретическая справка**

- **Что это?** Типизированный интерфейс логирования, предоставляющий методы для записи сообщений с различными уровнями важности и автоматической категоризацией по типу-источнику.
- **Зачем это?** Обеспечивает единообразие формата логов, упрощает фильтрацию и агрегацию в системах мониторинга, позволяет динамически управлять уровнем детализации без изменения кода.
- **Как это реализовано?** `ILogger<T>` резолвится через `ILoggerFactory`. Тип `T` преобразуется в строку-категорию (`Namespace.Type`). Каждый вызов метода лога создает `LogRecord` с уровнем, сообщением, исключением и контекстными данными. Провайдеры (Console, Debug, EventSource) форматируют запись согласно глобальным настройкам и выводят в целевое устройство.
- **Экосистема и альтернативы**

| Интерфейс логирования     | Типизация                        | Область применения                              |
| ------------------------- | -------------------------------- | ----------------------------------------------- |
| `ILogger<T>` (встроенный) | Высокая (DI-интеграция, уровни)  | ✅ Базовая инфраструктура, консоль/микросервисы |
| `Console.WriteLine`       | Отсутствует                      | ❌ Отладка, не для production                   |
| `Serilog.ForContext<T>`   | Высокая + расширенные обогащения | ❌ Enterprise-стеки, сложные пайплайны          |
| `EventSource`             | Низкая (бинарный, трассировка)   | ❌ Профилирование, низкоуровневая диагностика   |

## Секция 2: Message Templates — структурированные параметры

### Задача 7.2: Реализовать логирование через message templates вместо строковой интерполяции

**Описание:** Понять разницу между `ILogger` template `{UserId}` и строковой интерполяцией `$"..."`. Научиться передавать параметры как именованные аргументы для сохранения структуры. Убедиться, что при JSON-форматировании параметры становятся отдельными полями.

**Способы достижения:**

- Заменить интерполяцию: `logger.LogInformation($"User {id} logged")` → `logger.LogInformation("User {UserId} logged", id)`
- Добавить несколько параметров: `logger.LogInformation("User {UserId} from {Ip} accessed {Resource}", id, ip, resource)`
- Настроить временный JSON-форматтер и проверить вывод: поля `UserId`, `Ip`, `Resource` должны быть отдельными ключами
- Закоммитить: `git commit -am "feat(module7): use structured message templates"`

**Результат:** JSON-лог с отдельными полями `"UserId": 123`, `"Ip": "10.0.0.1"`, коммит с переходом на template-логирование.

📚 **Теоретическая справка**

- **Что это?** Синтаксис шаблонизированных сообщений, где параметры передаются как именованные аргументы, а не встраиваются в строку.
- **Зачем это?** Сохраняет структурированность данных для агрегаторов (ELK, Seq), позволяет индексировать параметры отдельно от текста, снижает аллокации строк за счёт отложенного форматирования.
- **Как это реализовано?** Компилятор или runtime-парсер анализирует шаблон, выделяет плейсхолдеры `{Name}` и сопоставляет их с позиционными аргументами. При использовании `JsonConsoleFormatter` параметры сериализуются как отдельные ключи объекта. Форматирование в человекочитаемую строку происходит только при выводе в текстовый провайдер.
- **Экосистема и альтернативы**

| Стиль логирования                   | Структурированность      | Производительность                  |
| ----------------------------------- | ------------------------ | ----------------------------------- |
| `MessageTemplate` (`{Param}`)       | Высокая (отдельные поля) | Высокая (отложенное форматирование) |
| Строковая интерполяция (`$"{val}"`) | Низкая (всё в строке)    | Средняя (аллокация строки сразу)    |
| `string.Format`                     | Низкая                   | Низкая (рефлексия, аллокации)       |
| Анонимные объекты в `BeginScope`    | Высокая                  | Средняя (доп. аллокации для скоупа) |

## Секция 3: Logging Scopes — контекстная группировка

### Задача 7.3: Реализовать логические группы событий через BeginScope

**Описание:** Изучить механизм `ILogger.BeginScope()` для добавления контекста ко всем логам внутри блока. Понять, что scope идеален для отслеживания операций/транзакций. Реализовать scope с уникальным идентификатором для группировки связанных событий.

**Способы достижения:**

- Обернуть блок кода: `using (logger.BeginScope(new { TransactionId = Guid.NewGuid() })) { ... }`
- Внутри scope добавить несколько логов и убедиться, что `TransactionId` присутствует в каждом
- Проверить вложенные scope: контекст должен накапливаться, не перезаписываться
- Закоммитить: `git commit -am "feat(module7): implement logging scopes for operation tracking"`

**Результат:** Два и более JSON-лога с одинаковым `TransactionId` внутри scope, коммит с реализацией контекстной группировки.

📚 **Теоретическая справка**

- **Что это?** Механизм временного расширения контекста логирования, добавляющий набор ключей-значений ко всем записям в пределах области видимости.
- **Зачем это?** Позволяет группировать логи по операциям, транзакциям или сессиям без явной передачи параметров в каждый вызов, упрощает отладку распределённых сценариев.
- **Как это реализовано?** `BeginScope` создает объект-контекст и помещает его в `AsyncLocal<Stack<object>>`. Все последующие вызовы `Log` обходят стек скоупов и включают их данные в `LogRecord`. При выходе из `using` скоуп удаляется из стека. Вложенные скоупы накапливаются: каждый новый добавляет поля поверх существующих.
- **Экосистема и альтернативы**

| Механизм контекста         | Время жизни                   | Область применения                      |
| -------------------------- | ----------------------------- | --------------------------------------- |
| `BeginScope` (stack-based) | Блок кода / асинхронный поток | ✅ Операции, транзакции, сессии         |
| Глобальные обогащения      | Весь цикл приложения          | ❌ Статичные поля (версия, хост)        |
| Явная передача параметров  | Локальный вызов               | ❌ Многословно, не масштабируется       |
| `Activity.Baggage`         | Асинхронный контекст          | ❌ Сквозная трассировка, не только логи |

## Секция 4: Интеграция JsonConsoleFormatter

### Задача 7.4: Настроить глобальный JSON-форматтер для консольного вывода

**Описание:** Заменить текстовый вывод логгера на структурированный JSON через `JsonConsoleFormatter`. Настроить опции форматирования: отступы, включение скоупов, формат временных меток. Убедиться, что все логи приложения, включая системные, выводятся в валидном JSON.

**Способы достижения:**

- В `Program.cs` настроить: `builder.Logging.ClearProviders().AddJsonConsole(opt => opt.JsonWriterOptions.Indented = true)`
- Запустить `dotnet run` и проверить, что каждый выводимый лог — валидный JSON-объект
- Убедиться, что поля `Timestamp`, `LogLevel`, `Category`, `Message`, параметры шаблона — все присутствуют
- Закоммитить: `git add .`
  `git commit -m "feat(module7): configure JsonConsoleFormatter globally"`

**Результат:** Валидный JSON-вывод всего приложения.

📚 **Теоретическая справка**

- **Что это?** Провайдер логирования, сериализующий `LogRecord` в структурированный JSON вместо текстового шаблона.
- **Зачем это?** Обеспечивает машинно-читаемый формат для парсинга агрегаторами, сохраняет типизацию полей, упрощает интеграцию с системами централизованного логирования без дополнительных преобразований.
- **Как это реализовано?** `JsonConsoleFormatter` реализует `ConsoleFormatter` и использует `Utf8JsonWriter` для потоковой сериализации. При вызове `Write` он последовательно записывает обязательные поля (`Timestamp`, `LogLevel`, `Category`), затем параметры шаблона, данные скоупов и исключение. Опции `JsonWriterOptions` контролируют форматирование (отступы, кодировка).
- **Экосистема и альтернативы**

| Форматтер                      | Вывод                       | Область применения                       |
| ------------------------------ | --------------------------- | ---------------------------------------- |
| `JsonConsoleFormatter`         | Структурированный JSON      | ✅ Консоль, агрегация, CI/CD             |
| `SimpleConsoleFormatter`       | Текстовый, человекочитаемый | ❌ Локальная разработка, отладка         |
| `SystemdConsoleFormatter`      | Журнальный формат systemd   | ❌ Linux-демоны, интеграция с journald   |
| `Serilog.CompactJsonFormatter` | Compact JSON (одно строка)  | ❌ Высоконагруженные продакшен-пайплайны |

## Секция 5: Диагностика через dotnet-trace

### Задача 7.5: Настроить сбор диагностических событий без изменения кода

**Описание:** Изучить инструмент `dotnet-trace` для сбора событий логирования в рантайме. Понять провайдер `Microsoft-Extensions-Logging` и формат `nettrace`. Научиться конвертировать трассировку для визуального анализа в Speedscope.

**Способы достижения:**

- Запустить приложение в фоне: `dotnet run &`
- Собрать трассировку: `dotnet trace collect --process-id <pid> --providers Microsoft-Extensions-Logging`
- Конвертировать и открыть в `speedscope`: `dotnet trace convert trace.nettrace`
- Закоммитить артефакты: `git add diagnostics/`
  `git commit -m "chore(module7): add diagnostic tooling setup"`

**Результат:** Файл `trace.nettrace` и скриншот из Speedscope с таймлайном лог-событий, коммит с настройкой диагностики.

📚 **Теоретическая справка**

- **Что это?** CLI-утилита для сбора EventPipe-событий из запущенного .NET-процесса без внедрения кода или перезапуска.
- **Зачем это?** Позволяет диагностировать production-инциденты с минимальным оверхедом, анализировать производительность и поведение системы постфактум, не требуя предварительной инструментации.
- **Как это реализовано?** `dotnet-trace` подключается к целевому процессу через EventPipe, подписывается на провайдеры (например, `Microsoft-Extensions-Logging`), буферизует события в бинарный формат `.nettrace`. Конвертер преобразует трассировку в формат Chrome Trace или Speedscope для визуализации временных диаграмм и вызовов.
- **Экосистема и альтернативы**

| Инструмент диагностики               | Режим сбора        | Область применения                                  |
| ------------------------------------ | ------------------ | --------------------------------------------------- |
| `dotnet-trace`                       | Runtime, EventPipe | ✅ Production-диагностика, без кода                 |
| `dotnet-counters`                    | Runtime, метрики   | ❌ Мониторинг ресурсов, не логи                     |
| `dotnet-gcdump`                      | Снимок кучи        | ❌ Анализ утечек памяти                             |
| Application Insights / OpenTelemetry | Интеграция в код   | ❌ Требует инструментации, но даёт больше контекста |
