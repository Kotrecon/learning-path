# Модуль 2 — DI & Lifetime Management (внедрение зависимостей и управление временем жизни)

> Цель: Научиться управлять временем жизни зависимостей так, чтобы объекты создавались, использовались и освобождались предсказуемо.

---

## Вступление

Этот модуль посвящён тому, как в .NET-приложении управлять временем жизни объектов так, чтобы зависимости создавались, использовались и освобождались предсказуемо. DI (внедрение зависимостей) — это способ передавать сервисы через `контейнер`, `lifetime` (время жизни) — правило, как долго объект существует, `scope` (область видимости) — граница, внутри которой сервис считается общим, а `Dispose` — метод для детерминированного освобождения ресурсов.

---

## Секция 1: Регистрация сервисов в контейнере

**В этой секции рассматривается базовый механизм DI-контейнера .NET — регистрация сервисов в `IServiceCollection`. Контейнер управляет созданием объектов и их жизненным циклом через стратегии `Singleton`, `Scoped` и `Transient`.**

📚 **Теоретическая справка**

- **Что это?**  
    Механизм внедрения зависимостей (Dependency Injection), при котором контракт и его реализация регистрируются в контейнере `IServiceCollection`, а контейнер управляет созданием объекта и его жизненным циклом.

- **Зачем это?**  
    Такой подход снижает связанность между компонентами, делает архитектуру более гибкой, упрощает модульное тестирование и позволяет централизованно управлять составом зависимостей приложения.

- **Как это реализовано?**  
    Контейнер хранит набор дескрипторов `ServiceDescriptor`, где зафиксированы тип сервиса, реализация и стратегия lifetime. При вызове `GetRequiredService<T>` контейнер определяет, нужно ли вернуть уже существующий экземпляр, создать новый или привязать объект к текущему scope. Создание происходит через фабрику, рефлексию или внутренние compiled delegates, а вложенные зависимости резолвятся рекурсивно.

- **Что важно понимать?**  
    Lifetime — это не просто способ создания объекта, а правило его существования в системе. Неправильный выбор lifetime часто приводит к трудноуловимым багам: лишнему удержанию памяти, неправильному переиспользованию состояния и нарушению потоковой безопасности.

- **Границы применимости**
    - `AddSingleton` — для общих, дорогих или неизменяемых зависимостей, которые должны жить всё время работы приложения.
    - `AddScoped` — для зависимых, которые должны быть общими только внутри одной единицы работы.
    - `AddTransient` — для лёгких stateless-объектов, которые не должны сохранять состояние между вызовами.
    - Ручное создание через `new` обычно уместно только для локальных объектов без внешних зависимостей.

- **Важная оговорка**  
    Одна из самых опасных ошибок — `captive dependency (захваченная зависимость)`: когда `Singleton` удерживает зависимость, рассчитанную на короткую жизнь, например `Scoped`. В таком случае объект начинает жить дольше, чем должен, и это ломает саму модель lifetime.

- **Экосистема и альтернативы**

| Стратегия                                          | Поведение                                           | Область применения                                                 |
| -------------------------------------------------- | --------------------------------------------------- | ------------------------------------------------------------------ |
| `AddSingleton`                                     | Один экземпляр на время жизни приложения            | Кеши, конфигурации, тяжёлые сервисы, логгеры                       |
| `AddScoped`                                        | Один экземпляр на область видимости (scope, скоуп)  | БД-контексты, транзакции, unit of work (единица работы)            |
| `AddTransient`                                     | Новый экземпляр при каждом запросе                  | Лёгкие утилиты, stateless-обработчики (обработчики без состояния)  |
| Ручное `new` / Service Locator (локатор сервисов)  | Прямая инициализация или глобальный поиск           | ❌ Антипаттерн: жёсткая связность, сложность тестирования          |

- **Типичные ошибки**
    - Регистрируют сервисы без понимания разницы между lifetime.
    - Используют `Singleton` для объектов со скрытым состоянием.
    - Путают `Scoped` и `Transient`.
    - Создают зависимости внутри бизнес-логики вместо явной регистрации.
    - Применяют service locator вместо прозрачной инъекции.
    - Не замечают `captive dependency`, когда lifetime объектов конфликтует между собой.

- **Production-практика**
    - Регистрация должна быть явной и централизованной.
    - Для тяжёлых и общих ресурсов использовать `Singleton`.
    - Для запрос-ориентированных зависимостей использовать `Scoped`.
    - Для короткоживущих stateless-объектов использовать `Transient`.
    - Перед финальной сдачей проверять, не нарушает ли схема регистрации границы жизненного цикла.
    - В Development .NET 10 по умолчанию включает `ValidateScopes=true` и `ValidateOnBuild=true`. Это даёт ранний фейл при captive dependency. В Production валидацию часто отключают (`ValidateScopes=false`) для экономии времени старта, поэтому ошибка может всплыть только при рантайм-резолве.

## Секция 2: Ручное создание скоупов в консоли

**Дальше конфигурация поднимается на уровень управления областью видимости: сначала базовая регистрация сервисов, а затем — явное создание scope в консольных приложениях.**

📚 **Теоретическая справка**

- **Что это?**  
    Явное управление областью видимости зависимостей в приложениях, где контейнер не создаёт scope автоматически по запросу, как это происходит в веб-приложениях.

- **Зачем это?**  
    В консольном приложении всё обычно начинается с root provider, и если не создать scope вручную, `Scoped`-сервисы фактически живут слишком долго или используются вне предполагаемой границы работы. Это особенно опасно в циклах, воркерах и batch-процессах.

- **Как это реализовано?**  
    `IServiceScopeFactory` создаёт дочерний `IServiceProvider`, внутри которого резолвятся scoped-зависимости. Всё, что создано в пределах `using`, связано с этим scope, а при выходе из блока scope уничтожается, и контейнер освобождает все отслеживаемые объекты.

- **Что важно понимать?**  
    Scope — это не "дополнительная формальность", а архитектурная граница. Он должен соответствовать реальной единице работы: обработке одного сообщения, одного файла, одной команды или одной итерации бизнес-цикла.

- **Границы применимости**
    - `IServiceScopeFactory.CreateScope()` — основной способ для консольных приложений, воркеров и batch-обработки.
    - Автоматический HTTP-scope существует только в веб-сценариях.
    - `BackgroundService.ExecuteAsync()` полезен для фоновых задач, но не заменяет явный контроль scope в произвольной консольной логике.
    - `IServiceProvider.CreateScope()` технически работает, но `IServiceScopeFactory` обычно лучше выражает намерение.

- **Типичные ошибки**
    - Не создают scope в консольном цикле.
    - Используют `Scoped`-сервисы вне `using`.
    - Держат scope слишком долго.
    - Путают scope с временем жизни всего приложения.
    - Резолвят сервис один раз и потом переиспользуют его как singleton.
    - Создают один огромный scope на весь процесс вместо scope на операцию.

- **Production-практика**
    - В batch-обработке и воркерах scope должен создаваться на единицу работы.
    - Scoped-сервисы не должны переживать свой контекст.
    - Scope нужно закрывать детерминированно через `using`.
    - Если внутри цикла есть ресурсы или внешние зависимости, scope должен совпадать с циклом обработки.

## Секция 3: Верификация поведения `IDisposable`

**После управления scope логично перейти к детерминированному освобождению ресурсов. Здесь контейнер берёт на себя вызов `Dispose` для объектов, которые он создал.**

📚 **Теоретическая справка**

- **Что это?**  
    Интеграция механизма детерминированного освобождения ресурсов с DI-контейнером, который отслеживает созданные им объекты и вызывает `Dispose` в нужный момент.

- **Зачем это?**  
    Это необходимо, чтобы вовремя освобождать файлы, сокеты, соединения, хендлы ОС и другие ресурсы, которые нельзя полагать на сборщик мусора. Без явного освобождения приложение может продолжать работать, но уже деградировать по памяти и по внешним ресурсам.

- **Как это реализовано?**  
    Если контейнер создал объект, реализующий `IDisposable` или `IAsyncDisposable`, он может взять на себя его освобождение. Для `Singleton` `Dispose` вызывается при завершении хоста, для `Scoped` — при закрытии scope. Для `Transient` освобождение зависит от того, кто именно владеет объектом и кто обязан его освободить.

- **Что важно понимать?**  
    `Dispose` — это не про "очистку памяти", а про завершение владения ресурсом. GC не гарантирует своевременное закрытие внешних ресурсов, а значит, без `Dispose` приложение может столкнуться с исчерпанием соединений, блокировками файлов или накоплением неосвобождённых handle-ов.

- **Границы применимости**
    - `IDisposable` нужен для синхронного освобождения.
    - `IAsyncDisposable` нужен, если освобождение само по себе асинхронное.
    - Финализатор — это запасной механизм, а не основа жизненного цикла.
    - `using` удобен для локального управления, но не заменяет DI там, где есть сложный граф зависимостей.

- **Типичные ошибки**
    - Рассчитывают, что контейнер сам вызовет `Dispose` у `Transient`-сервисов. Да, контейнер может это делать, но создаёт накладные расходы на память для трекинга каждого экземпляра. В production `Transient` должен быть строго stateless, а ресурсы освобождаться вручную.
    - Не реализуют `Dispose` у объектов, которые владеют ресурсами.
    - Держат unmanaged-ресурсы в `Singleton` без корректного shutdown-пути.
    - Путают финализатор с нормальным управлением жизненным циклом.
    - Оставляют ресурсы открытыми при аварийной остановке.

- **Production-практика**
    - Все ресурсы, которые имеют жизненный цикл, должны освобождаться детерминированно.
    - Если сервис владеет соединением, потоком или файловым handle, он обязан поддерживать `Dispose` или `DisposeAsync`.
    - Для `Singleton` нужен ясный shutdown-путь.
    - Для `Scoped` освобождение должно происходить при закрытии scope.

## Секция 4: Интеграционный тест жизненного цикла

**Когда lifecycle уже настроен, важно не просто надеяться на правильную работу, а зафиксировать её проверкой. Эта секция связывает DI-контейнер с повторяемым тестом.**

📚 **Теоретическая справка**

- **Что это?**  
    Сценарий проверки архитектурных соглашений в реальном runtime, где приложение многократно создаёт scope, резолвит зависимости и наблюдает, как контейнер их освобождает.

- **Зачем это?**  
    Такой тест подтверждает, что жизненный цикл объектов работает не только "на бумаге", но и при повторяемом запуске. Это помогает обнаружить утечки ссылок, ошибки в scope-границах и неправильное освобождение ресурсов.

- **Как это реализовано?**  
    Внутри цикла создаётся `IServiceScope`, внутри него резолвится `Scoped`-сервис, после чего scope завершается через `using`. Это заставляет контейнер закрыть все объекты, связанные с этим scope, и позволяет наблюдать фактический порядок создания и уничтожения экземпляров.

- **Что важно понимать?**  
    Интеграционный тест проверяет именно поведение системы в динамике. Статическая проверка регистраций не способна показать, что происходит после выхода из scope, а именно там обычно и проявляются ошибки жизненного цикла.

- **Экосистема и альтернативы**

| Метод валидации                                 | Уровень изоляции          | Сложность интеграции                                    |
| ----------------------------------------------- | ------------------------- | ------------------------------------------------------- |
| Runtime-цикл с `using scope`                    | Высокий (полный процесс)  | ✅ Низкая, нативная консольная проверка                 |
| `xUnit` + `WebApplicationFactory`               | Высокий (в памяти)        | ❌ Требует мок-хоста, избыточно для консоли             |
| Статическая валидация на старте                 | Низкий                    | ❌ Не проверяет `Dispose` и runtime-поведение           |
| BenchmarkDotNet                                 | Средний                   | ❌ Сфокусировано на производительности, не на lifecycle |

- **Типичные ошибки**
    - Проверяют только регистрацию, но не жизненный цикл.
    - Не повторяют тест несколько раз подряд.
    - Не фиксируют логи создания и уничтожения экземпляров.
    - Смешивают статическую и runtime-валидацию.
    - Игнорируют накопление ресурсов в цикле.

- **Production-практика**
    - Интеграционный тест должен быть повторяемым.
    - Логи создания и уничтожения объектов должны быть однозначными.
    - Проверка lifecycle должна входить в обычный набор регрессионных тестов.
    - Чем ближе тест к реальному сценарию обработки, тем выше его ценность.

## Секция 5: Мониторинг ресурсов через `dotnet-counters`

**После тестирования логично перейти к наблюдению за реальным поведением приложения в runtime. Здесь конфигурация превращается в наблюдаемые метрики.**

📚 **Теоретическая справка**

- **Что это?**  
    Диагностика работы CLR через системные счётчики, позволяющая наблюдать за памятью, GC и другими метриками в реальном времени без изменения кода приложения.

- **Зачем это?**  
    Это даёт возможность отличить реальную утечку памяти от нормального поведения сборщика мусора и понять, освобождаются ли ресурсы после завершения scope.

- **Как это реализовано?**  
    Инструмент подключается к процессу через EventPipe и читает метрики из `System.Runtime`, включая размер GC-кучи, количество сборок и объём аллокаций. По этим данным можно увидеть тренд потребления памяти и сопоставить его с поведением scope-ов.

- **Что важно понимать?**  
    Одного значения GC Heap Size недостаточно для вывода об утечке. Важен повторяемый сценарий: если приложение многократно проходит один и тот же цикл, а память всё растёт и не стабилизируется, это уже сильный сигнал о проблеме.

- **Экосистема и альтернативы**

| Инструмент                       | Тип анализа              | Область применения                                         |
| -------------------------------- | ------------------------ | ---------------------------------------------------------- |
| `dotnet-counters`                | Реальное время           | ✅ Ранний индикатор утечки, быстрый запуск                 |
| `dotnet-gcdump`                  | Офлайн-анализ кучи       | ✅ Детальный анализ конкретной снимка кучи                 |
| Diagnostic Tools в IDE           | Интеграция в IDE         | ❌ Тяжелее, менее универсальны                             |
| BenchmarkDotNet                  | Микробенчмарки           | ❌ Сфокусировано на производительности, не на lifecycle    |

- **Типичные ошибки**
    - Смотрят только на один срез памяти.
    - Путают задержку GC с утечкой.
    - Не сопоставляют метрики с жизненным циклом scope.
    - Игнорируют тренд, глядя только на абсолютные числа.
    - Не проверяют, как ведёт себя память после завершения работы scope.

- **Production-практика**
    - Смотреть на память до и после серии одинаковых операций.
    - Сопоставлять метрики с логами создания и уничтожения объектов.
    - Использовать `dotnet-counters` как ранний индикатор проблемы.
    - Подтверждать отсутствие утечек не только логикой кода, но и фактическими метриками.

## Секция 6: Документация и сдача модуля

**Когда life-cycle уже настроен, проверен и наблюдается через метрики, остаётся зафиксировать контракт: почему выбрано именно такое решение и как это передаётся дальше в команду.**

📚 **Теоретическая справка**

- **Что это?**  
    Формализация инженерных решений в документации и процесс передачи изменений через Pull Request, где код и архитектурные решения проходят ревью до попадания в основную ветку.

- **Зачем это?**  
    Это делает изменения прозрачными, проверяемыми и воспроизводимыми. Для темы DI это особенно важно, потому что lifetime-ошибки часто не видны сразу и требуют контекста, чтобы понять, почему было выбрано именно такое решение.

- **Как это реализовано?**  
    Git хранит diff между состояниями веток, PR связывает изменения с обсуждением и CI-проверками, а история коммитов позволяет отследить, как именно развивалось решение. Хорошие сообщения коммитов помогают автоматизировать changelog и сопровождение проекта.

- **Что важно понимать?**  
    Документация здесь фиксирует не только "что сделано", но и "почему так сделано". Для DI и lifetime это критично, потому что ошибка в архитектурной границе может проявиться только в production, а не на уровне локального запуска.

- **Экосистема и альтернативы**

| Практика сдачи                 | Механизм                                               | Отличия от задачи                                      |
| ------------------------------ | ------------------------------------------------------ | ------------------------------------------------------ |
| PR с документацией решений     | Изолированное ревью, CI-валидация, исторический аудит  | ✅ Используется                                        |
| Direct commit в main           | Мгновенный деплой без ревью                            | ❌ Высокий риск поломки DI-контейнера и lifecycle      |
| Squash merge                   | Атомарное объединение коммитов                         | ❌ Теряется пошаговая история принятия решений         |
| Email- или issue-based review  | Ручное описание изменений                              | ❌ Не масштабируется, нет привязки к коду и CI         |

- **Типичные ошибки**
    - Делают PR без объяснения архитектурного смысла.
    - Не прикладывают доказательства проверки.
    - Пишут коммиты без ясной структуры.
    - Сливают изменения без CI.
    - Не документируют принятые решения.

- **Production-практика**
    - Любое изменение DI и точки входа должно сопровождаться пояснением.
    - Коммиты должны быть пригодны для автоматической обработки.
    - PR должен отвечать на вопросы: что изменилось, почему и как проверено.
    - Для серьёзных архитектурных решений полезно фиксировать краткий ADR.

## Критерий приёмки

- `AddSingleton`/`AddScoped`/`AddTransient` зарегистрированы корректно для сценария приложения.
- В консольном приложении создан ручной `IServiceScope` через `IServiceScopeFactory`.
- `Scoped`-сервис освобождается при выходе из `using scope`.
- Интеграционный тест несколько раз создаёт и закрывает scope, логируя создание/уничтожение.
- `dotnet-counters` показывает стабильную память после серии операций (нет роста GC Heap Size).
- Все изменения зафиксированы в Git с понятной историей коммитов и PR с пояснением решений.
