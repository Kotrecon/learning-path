## Секция 1: Регистрация сервисов в контейнере

### Задача 2.1: Зарегистрировать сервисы с разными стратегиями жизненного цикла

**Описание:** Изучить различия между AddSingleton, AddTransient и AddScoped. Зарегистрировать тестовые сервисы в контейнере через прямой доступ к `builder.Services` (паттерн `HostApplicationBuilder`). Убедиться, что сервисы успешно резолвятся через хост.

**Способы достижения:**

- Создать интерфейсы и реализации: `ILoggerService`/`ConsoleLogger`, `IDataProcessor`/`DataProcessor`
- В `Program.cs` добавить регистрацию: `builder.Services.AddSingleton<ILoggerService, ConsoleLogger>()` и аналоги
- Резолвить сервисы: `host.Services.GetRequiredService<ILoggerService>()` и проверить создание экземпляра
- Закоммитить:
  `git add .`
  `git commit -m "feat(module2): implement basic DI registration"`

**Результат:** Код регистрации в Program.cs, успешный резолв всех сервисов, коммит с DI-настройкой в репозитории.

## Секция 2: Ручное создание скоупов в консоли

### Задача 2.2: Реализовать создание скоупов через IServiceScopeFactory

**Описание:** Понять, что консольное приложение не создаёт скоупы автоматически. Использовать IServiceScopeFactory для явного создания области видимости. Проверить, что каждый скоуп создаёт новый экземпляр Scoped-сервиса.

**Способы достижения:**

- Получить фабрику: `var scopeFactory = host.Services.GetRequiredService<IServiceScopeFactory>()`
- Создать скоуп: `using var scope = scopeFactory.CreateScope()` и резолвить внутри
- Добавить логирование ID экземпляра для подтверждения создания нового объекта
- Закоммитить: `git commit -am "feat(module2): implement manual scope creation"`

**Результат:** Код создания скоупа, лог с уникальными ID экземпляров для каждого скоупа, коммит в истории репозитория.

## Секция 3: Верификация поведения IDisposable

### Задача 2.3: Проверить вызов Dispose для разных стратегий жизненного цикла

**Описание:** Реализовать интерфейс IDisposable в тестовом сервисе. Зарегистрировать сервис как Singleton, Scoped и Transient. Наблюдать, когда контейнер вызывает Dispose для каждой стратегии.

**Способы достижения:**

- Добавить `: IDisposable` к сервису и реализовать метод Dispose с логированием
- Зарегистрировать три экземпляра с разными стратегиями, резолвить и наблюдать за логами
- Остановить хост (Ctrl+C) и зафиксировать порядок вызовов Dispose
- Закоммитить: `git commit -am "feat(module2): verify Dispose behavior for lifetimes"`

**Результат:** Лог вызовов Dispose: Singleton — при остановке хоста, Scoped — при завершении скоупа, Transient — не вызывается, коммит с верификацией.

## Секция 5: Мониторинг ресурсов через dotnet-counters

### Задача 2.5: Настроить мониторинг памяти для выявления утечек

**Описание:** Использовать dotnet-counters для наблюдения за размером GC-кучи в реальном времени. Запустить приложение, создать несколько скоупов, убедиться, что память освобождается после завершения скоупов.

> ⚠️ Если инструмент не установлен: `dotnet tool install -g dotnet-counters`

**Способы достижения:**

- Запустить приложение в одном окне: `dotnet run`
- В другом окне: `dotnet-counters monitor --process-id <PID> System.Runtime`
- Наблюдать за метрикой `GC Heap Size` до и после создания скоупов
- Зафиксировать скриншот или лог, закоммитить: `git commit -am "chore(module2): add resource monitoring"`

**Результат:** Скриншот/лог dotnet-counters с подтверждением освобождения памяти, коммит с настройкой мониторинга в репозитории.
