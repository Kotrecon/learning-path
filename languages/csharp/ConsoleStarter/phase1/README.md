### 🔧 Готовые команды для быстрого старта (PowerShell, .NET 10)

```powershell
# 1. Создание структуры модулей
mkdir dotnet10-fundamentals && cd dotnet10-fundamentals
1..6 | ForEach-Object { mkdir "module$_" }

# 2. Базовые команды для каждого модуля
# В корне каждого модуля:
dotnet new console -n ModuleX -lang C# -f net10.0
dotnet add package Microsoft.Extensions.Hosting
dotnet add package TUnit  # для интеграционных тестов
dotnet build

# 3. Тест конфигурации (Модуль 3)
$env:App__Timeout="120"; dotnet run
dotnet run -- --App:Timeout=300

# 4. Parity-тест (Модуль 6) на TUnit
# В module6/:
dotnet build
dotnet test --filter "FullyQualifiedName~Parity"  # запуск parity-теста

# 5. Публикация (Модуль 6)
dotnet publish -c Release -r win-x64 --self-contained true -o ./publish
.\publish\Module6.exe  # Запуск без установленного .NET SDK
```

### 💡 Ключевые принципы .NET 10 для этой фазы

1. **`CreateApplicationBuilder` — новый стандарт**. Забудьте про `CreateDefaultBuilder`. Новый билдер даёт готовую инфраструктуру с меньшим бойлерплейтом.
2. **`IOptionsMonitor<T>` для hot-reload**. `IOptions<T>` кэшируется при первом запросе — это важно понимать, чтобы не ждать "магического" обновления.
3. **Скоупы в консоли — явно**. Нет автоматического скоупа на запрос, как в вебе. Используйте `IServiceScopeFactory` для единицы работы.
4. **Кооперативная отмена — часть дизайна**. `CancellationToken` должен быть в подписи `ExecuteAsync`, а проверка — в каждой итерации цикла.
5. **Baseline = воспроизводимость**. Парити-тесты, `dotnet publish`, структурированное логирование — это не опции, а требования к production-готовности.
