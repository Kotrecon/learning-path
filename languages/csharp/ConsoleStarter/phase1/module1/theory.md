# Модуль 1 — .NET 10 Console Modernization

> Цель: Освоить современную архитектуру консольного приложения .NET 10 на базе top-level statements, `Host.CreateApplicationBuilder(args)`, `builder.Services`, `builder.Configuration` и корректного `RunAsync()`.

---

## Вступление

Этот модуль задаёт базу для всего курса на .NET 10. Он показывает, как устроена современная точка входа консольного приложения, как `Program.cs` превращается в composition root и как приложение поднимается через Generic Host. [learn.microsoft](https://learn.microsoft.com/en-us/dotnet/core/extensions/generic-host)
В отличие от старых шаблонов с `class Program` и `Main`, здесь сразу используются top-level statements из C# 14, современный билдер и инфраструктура хоста, чтобы курс дальше плавно переходил к DI, lifetime и конфигурации. [anthonysimmon](https://anthonysimmon.com/dotnet-minimal-workers/)

## Секция 1: Top-level statements в C# 14

**В этой секции рассматривается современный синтаксис C# 14 — top-level statements. Он позволяет писать исполняемый код напрямую в `Program.cs` без явного объявления `class Program` и метода `Main`, делая старт приложения короче и понятнее.**

📚 **Теоретическая справка**

- **Что это?**  
  `Top-level statements` — синтаксис **C# 14**, позволяющий размещать исполняемый код напрямую в `.cs`-файле без явного объявления `class Program` и `static void Main`.

- **Зачем это?**  
  Снижает объём boilerplate-кода, делает точку входа компактнее и помогает сосредоточиться на запуске приложения, регистрациях и конфигурации.

- **Как это реализовано?**  
  Компилятор Roslyn преобразует top-level code в скрытый `internal class Program` с автоматически сгенерированным entry point. Для разработчика код выглядит как обычный последовательный блок, но для runtime это полноценная точка входа. Аргументы командной строки доступны как локальная переменная `args`. [learn.microsoft](https://learn.microsoft.com/en-us/dotnet/csharp/fundamentals/program-structure/top-level-statements)

- **Что важно понимать?**  
  Это не особый режим выполнения, а более компактная форма записи entry point. Top-level statements не меняют модель работы приложения, а только упрощают исходный код. [learn.microsoft](https://learn.microsoft.com/en-us/dotnet/csharp/fundamentals/program-structure/top-level-statements)

- **Экосистема и альтернативы**

| Подход                                 | Область применения                               | Отличия от задачи                                      |
| -------------------------------------- | ------------------------------------------------ | ------------------------------------------------------ |
| `class Program { static void Main() }` | Legacy-проекты и приложения с явной точкой входа | Требует больше служебного кода                         |
| `async Task Main(string[] args)`       | Консольные приложения с асинхронным запуском     | Даёт явный async-flow, но остаётся шаблонным           |
| C# scripts (`.csx`)                    | Прототипирование и локальные эксперименты        | Не рассчитаны на production-сборки                     |
| Top-level statements в **C# 14**       | Современные консольные приложения **.NET 10**    | ✅ Используется как стандартная компактная точка входа |

- **Типичные ошибки**
  - Смешивают bootstrap-логику и бизнес-логику в одном `Program.cs`.
  - Раздувают точку входа большим количеством кода.
  - Дублируют регистрацию инфраструктуры прямо в исполняемом блоке.
  - Не оставляют места для масштабирования точки входа.
  - Путают компактный синтаксис с отсутствием архитектуры.

- **Production-практика**
  - `Program.cs` должен оставаться кратким composition root.
  - Исполняемый код в точке входа должен только запускать приложение.
  - Если логика старта усложняется, её выносят в отдельные методы или классы.
  - Структура входа должна быть предсказуемой и читабельной.

## Секция 2: `Host.CreateApplicationBuilder(args)` в .NET 10

**Дальше приложение поднимается на уровень инфраструктуры: после компактного синтаксиса инициализируется современный билдер `Host.CreateApplicationBuilder(args)`, который является базовым способом старта консольного приложения в .NET 10.**

📚 **Теоретическая справка**

- **Что это?**  
  `Host.CreateApplicationBuilder(args)` — фабричный метод **.NET 10**, который создаёт `HostApplicationBuilder` с готовыми провайдерами конфигурации, логирования и DI-контейнером.

- **Зачем это?**  
  Он убирает ручную сборку инфраструктуры и сразу даёт доступ к `builder.Services`, `builder.Configuration`, `builder.Environment` и `builder.Logging`.

- **Как это реализовано?**  
  Билдер уже содержит базовую конфигурацию, включая JSON-файлы, переменные окружения и аргументы CLI. После регистрации сервисов вызывается `builder.Build()`, который создаёт `IHost` для запуска приложения. [anthonysimmon](https://anthonysimmon.com/dotnet-minimal-workers/)

- **Что важно понимать?**  
  `HostApplicationBuilder` — это не просто контейнер зависимостей, а единый механизм подготовки приложения к запуску. Именно через него удобно строить курс, потому что модули 2 и 3 опираются на `builder.Services` и `builder.Configuration`.

- **Экосистема и альтернативы**

| Компонент                             | Назначение                          | Статус в модуле                         |
| ------------------------------------- | ----------------------------------- | --------------------------------------- |
| `Host.CreateApplicationBuilder(args)` | Современный builder для **.NET 10** | ✅ Используется                         |
| `Host.CreateDefaultBuilder(args)`     | Старый паттерн Generic Host         | ❌ Не используется в курсе              |
| `WebApplication.CreateBuilder(args)`  | Веб-хост для ASP.NET Core           | ❌ Не рассматривается                   |
| `BackgroundService`                   | Базовый тип фонового сервиса        | ❌ Используется позже, не в этом модуле |

- **Типичные ошибки**
  - Используют старый `Host.CreateDefaultBuilder(args)` вместо современного билдера **.NET 10**.
  - Путают `HostApplicationBuilder` и `IHost`.
  - Смешивают конфигурацию и бизнес-логику в `Program.cs`.
  - Не используют доступные сразу `builder.Services` и `builder.Configuration`.
  - Ожидают, что инфраструктура появится без `Build()`.

- **Production-практика**
  - Создавать хост через `Host.CreateApplicationBuilder(args)`.
  - Регистрировать всё инфраструктурное через `builder.Services`.
  - Читать и дополнять конфигурацию через `builder.Configuration`.
  - Использовать `builder.Environment` для средовых различий.
  - Держать стартовую инфраструктуру одинаковой во всех консольных проектах.

## Секция 3: `RunAsync()` и graceful shutdown в .NET 10

**После инициализации хоста логично перейти к управлению его жизненным циклом. Здесь используется `await host.RunAsync()` как стандартный механизм длительного и корректного выполнения приложения.**

📚 **Теоретическая справка**

- **Что это?**  
  `RunAsync()` — асинхронный метод запуска хоста **.NET 10**, который удерживает процесс активным до сигнала завершения и обеспечивает корректную остановку.

- **Зачем это?**  
  Он нужен для консольных и фоновых приложений, где важны корректное завершение, освобождение ресурсов и предсказуемый shutdown-path.

- **Как это реализовано?**  
  `RunAsync()` подписывается на системные сигналы завершения, включая `Console.CancelKeyPress`, `SIGINT` и `SIGTERM`. При получении сигнала хост инициирует остановку и ждёт завершения фоновых компонентов. [learn.microsoft](https://learn.microsoft.com/en-us/dotnet/core/extensions/generic-host)

- **Что важно понимать?**  
  Корректное завершение — часть архитектуры приложения. В **.NET 10** нельзя рассматривать shutdown как второстепенную деталь, потому что от него зависят целостность данных и стабильность процесса.

- **Экосистема и альтернативы**

| Метод                                     | Поведение                                  | Отличия от задачи                     |
| ----------------------------------------- | ------------------------------------------ | ------------------------------------- |
| `await host.RunAsync()`                   | Асинхронный запуск с graceful shutdown     | ✅ Используется                       |
| `host.Run()`                              | Синхронная блокировка                      | ❌ Менее удобен для async-flow        |
| `StartAsync()` + `WaitForShutdownAsync()` | Раздельное управление запуском и ожиданием | ❌ Слишком вручную для базового курса |
| `StopAsync()`                             | Программная остановка                      | ❌ Это не основной способ запуска     |

- **Типичные ошибки**
  - Игнорируют сигналы завершения приложения.
  - Используют долгие операции без `CancellationToken`.
  - Не освобождают ресурсы при остановке.
  - Путают запуск приложения и рабочую бизнес-логику.
  - Пишут фоновые сценарии без пути завершения.

- **Production-практика**
  - Всегда проектировать корректный shutdown-path.
  - Использовать `CancellationToken` в длительных операциях.
  - Проверять поведение приложения при остановке.
  - Держать жизненный цикл приложения управляемым и предсказуемым.

## Секция 4: `Program.cs` как composition root в .NET 10

**Когда синтаксис и хост уже настроены, важно зафиксировать правильную архитектуру: `Program.cs` в .NET 10 должен быть composition root, а не местом для бизнес-логики.**

📚 **Теоретическая справка**

- **Что это?**  
  Composition root — точка, где приложение собирается из инфраструктурных частей: DI, конфигурации, логирования и запуска.

- **Зачем это?**  
  Такой подход делает код понятным, уменьшает связность и упрощает дальнейшее расширение приложения в модулях 2 и 3.

- **Как это реализовано?**  
  В `Program.cs` остаются только создание билдера, регистрация сервисов, сборка хоста и запуск через `RunAsync()`. Бизнес-логика выносится в отдельные классы и сервисы.

- **Что важно понимать?**  
  Минимализм `Program.cs` не означает отсутствие архитектуры. Наоборот, это признак того, что архитектура вынесена в правильные слои.

- **Экосистема и альтернативы**

| Архитектура точки входа                | Сложность | Область применения                          |
| -------------------------------------- | --------- | ------------------------------------------- |
| Минимальный `Program.cs` в **.NET 10** | Низкая    | ✅ Консольные приложения, worker-сервисы    |
| `Program.cs` + `Startup.cs`            | Средняя   | ❌ Старый шаблон, не нужен в курсе          |
| Отдельный фабричный слой               | Высокая   | ❌ Для очень сложной условной инициализации |
| Composition root в отдельной сборке    | Высокая   | ❌ Для крупных enterprise-решений           |

- **Типичные ошибки**
  - Перегружают `Program.cs` лишней логикой.
  - Дублируют регистрацию зависимостей в нескольких местах.
  - Смешивают инфраструктуру и domain layer.
  - Держат запуск и выполнение бизнес-операций в одном блоке.
  - Путают краткость файла с отсутствием архитектурного слоя.

- **Production-практика**
  - Держать `Program.cs` коротким и прозрачным.
  - Выносить сложную сборку зависимостей в отдельные методы.
  - Поддерживать единый стиль запуска во всех проектах.
  - Сохранять только orchestration старта в точке входа.

## Секция 5: Диагностика сборки и зависимостей в .NET 10

**После настройки архитектуры важно не просто надеяться на корректную сборку, а проверить инструменты и зависимости, которые использует проект `.NET 10`.**

📚 **Теоретическая справка**

- **Что это?**  
  Набор команд `dotnet` для диагностики SDK, сборки и пакетов: `dotnet --info`, `dotnet build -v:d`, `dotnet list package`.

- **Зачем это?**  
  Это помогает отслеживать версию SDK **.NET 10**, видеть граф зависимостей и проверять воспроизводимость сборки.

- **Как это реализовано?**  
  `dotnet --info` показывает установленные SDK и рантаймы. `dotnet build -v:d` даёт подробный лог MSBuild. `dotnet list package` отображает прямые и транзитивные пакеты проекта.

- **Что важно понимать?**  
  Диагностика должна быть частью обычного workflow, а не реакцией на сбой. Для курса **.NET 10** это особенно полезно, потому что помогает держать окружение и зависимости под контролем.

- **Экосистема и альтернативы**

| Инструмент                                      | Функционал                          | Отличия от задачи                            |
| ----------------------------------------------- | ----------------------------------- | -------------------------------------------- |
| `dotnet --info` / `build -v:d` / `list package` | Базовая CLI-диагностика **.NET 10** | ✅ Используется                              |
| `dotnet nuget locals`                           | Управление кэшем пакетов            | ❌ Не анализирует граф зависимостей          |
| `dotnet-outdated-tool`                          | Поиск обновлений пакетов            | ❌ Внешний инструмент, не базовый для модуля |
| `MSBuildStructuredLog`                          | Визуализация сборки                 | ❌ Требует отдельного binlog-процесса        |

- **Типичные ошибки**
  - Не проверяют SDK перед стартом работы.
  - Игнорируют транзитивные зависимости.
  - Не сохраняют сведения о сборке.
  - Диагностируют только при падении CI.
  - Не контролируют версию .NET 10 в окружении.

- **Production-практика**
  - Проверять `dotnet --info` перед работой и в CI.
  - Контролировать версии пакетов и SDK.
  - Использовать диагностику как регулярный шаг.
  - Фиксировать изменения графа зависимостей.

## Секция 6: Документация и сдача модуля в .NET 10

**Когда базовая архитектура уже настроена, остаётся зафиксировать решения в документации и оформить сдачу через PR, чтобы модуль было легко проверять и сопровождать в `.NET 10`-курсе.**

📚 **Теоретическая справка**

- **Что это?**  
  Техническая документация и процесс сдачи через Pull Request, фиксирующие архитектурные решения и проверку результата.

- **Зачем это?**  
  Это даёт прозрачность, позволяет ревьюить изменения до интеграции и оставляет понятный след для команды.

- **Как это реализовано?**  
  Изменения вносятся в отдельной ветке, фиксируются Git-коммитами с понятными префиксами, а затем проходят PR-ревью и CI-проверки.

- **Что важно понимать?**  
  Для модуля **.NET 10** документация — не формальность, а часть инженерного качества.

- **Экосистема и альтернативы**

| Практика                     | Механизм                      | Отличия от задачи           |
| ---------------------------- | ----------------------------- | --------------------------- |
| Pull Request / Merge Request | Ревью, CI, история обсуждений | ✅ Используется             |
| Trunk-based development      | Прямые коммиты в main         | ❌ Не базовый формат курса  |
| Squash Merge                 | Сведение коммитов в один      | ❌ Теряет детализацию шагов |
| Issue-only review            | Обсуждение без Git-механики   | ❌ Плохо масштабируется     |

- **Типичные ошибки**
  - Делают PR без пояснения архитектурного смысла.
  - Не прикладывают проверяемые результаты.
  - Пишут нечитаемые коммиты.
  - Пропускают CI-проверки.
  - Не документируют важные решения.

- **Production-практика**
  - Любое изменение точки входа описывать в документации.
  - Коммиты оформлять в едином стиле.
  - PR должен отвечать на вопросы: что поменялось, почему и как проверено.
  - Для архитектурных решений оставлять краткий ADR.
