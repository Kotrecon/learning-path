# Модуль 1 — .NET 10 Console Modernization

## Секция 1: Переход на top-level statements (верхнеуровневые инструкции)

📚 **Теоретическая справка**

- **Что это?**  
  `Top-level statements (верхнеуровневые инструкции)` — синтаксис C# 9+, позволяющий размещать исполняемый код напрямую в `.cs`-файле без явного объявления `class Program` и метода `Main`.

- **Зачем это?**  
  Сокращает объём boilerplate (служебного кода), делает точку входа короче и читабельнее, снижает когнитивную нагрузку и помогает сосредоточиться на конфигурации, внедрении зависимостей и запуске приложения.

- **Когда это уместно?**  
  Подходит для консольных приложений, worker-сервисов, CLI-утилит, MVP и небольших инфраструктурных запусков. При сложной bootstrap-логике, множественных режимах старта или кастомной маршрутизации входа иногда удобнее оставить явный `Main`.

- **Как это реализовано?**  
  Компилятор Roslyn преобразует top-level code (верхнеуровневый код) в скрытый `internal class Program` с автоматически сгенерированным методом входа. Для разработчика код выглядит как свободный исполняемый блок, но для runtime (среды выполнения) это обычная точка входа с теми же возможностями, что и у классического `Main`. Аргументы командной строки доступны через локальную переменную `args`.

- **Что важно понимать?**  
  Это не “магия” и не особый режим выполнения, а более компактная форма записи entry point (точки входа). Подход не меняет runtime-модель приложения и не даёт производственных накладных расходов сам по себе.

- **Экосистема и альтернативы**

| Подход                                            | Область применения                                                                 | Отличия от задачи                                                              |
| ------------------------------------------------- | ---------------------------------------------------------------------------------- | ------------------------------------------------------------------------------ |
| `class Program { static void Main() }`            | Legacy-проекты, сложная точка входа                                                | Требует явного объявления, больше boilerplate (служебного кода)                |
| `async Task Main(string[] args)`                  | Console apps (консольные приложения) с I/O (вводом-выводом) и асинхронным запуском | Даёт явный async-flow (асинхронный поток выполнения), но сохраняет шаблонность |
| C# Scripts (`.csx`)                               | Прототипирование, REPL (интерактивная среда), локальные эксперименты               | Не рассчитаны на production-сборки и строгий lifecycle (жизненный цикл)        |
| Top-level statements (верхнеуровневые инструкции) | Современные консольные и worker-приложения                                         | ✅ Используется как компактная и стандартная точка входа                       |

- **Типичные ошибки**
  - Смешивают bootstrap-логику (логику запуска) и бизнес-логику в одном файле.
  - Раздувают `Program.cs` до нечитаемого блока.
  - Используют `Run()` вместо `RunAsync()` (асинхронного запуска) без необходимости.
  - Прячут конфигурацию и регистрацию сервисов прямо в исполняемом коде.
  - Не оставляют явной структуры для масштабирования точки входа.

- **Production-практика**
  - `Program.cs` должен быть composition root (корнем композиции), а не местом для бизнес-логики.
  - Инициализация должна быть короткой, предсказуемой и читаемой.
  - Если точка входа становится сложной, лучше выносить регистрацию и bootstrap в отдельные методы или классы.
  - Важно сохранять ясность shutdown-пути (пути корректного завершения) и конфигурации запуска.

## Секция 2: Инициализация IHost (универсального хоста)

📚 **Теоретическая справка**

- **Что это?**  
  `IHost (универсальный хост)` — абстракция, инкапсулирующая жизненный цикл приложения, контейнер внедрения зависимостей, систему конфигурации и логирования. `Host.CreateDefaultBuilder()` — фабричный метод для инициализации хоста с предустановленными соглашениями .NET.

- **Зачем это?**  
  Устраняет ручную настройку инфраструктуры, обеспечивает единообразие между типами приложений (консоль, фоновые задачи, микросервисы), гарантирует корректное управление ресурсами и согласованность конфигурационных источников.

- **Когда это уместно?**  
  Подходит для приложений, где нужен единый жизненный цикл: воркеры, фоновые службы, сервисы, консольные приложения с DI, а также проекты, где важны конфигурация, логирование и управляемый shutdown.

- **Как это реализовано?**  
  Фабрика последовательно добавляет провайдеры конфигурации (`appsettings.json`, переменные окружения, CLI-аргументы), регистрирует стандартные логгеры (`Console`, `Debug`), настраивает `IServiceCollection`. Вызов `.Build()` материализует контейнер DI, разрешает статические зависимости и переводит хост в состояние готовности. При старте активируются сервисы `IHostedService` в порядке регистрации.

- **Что важно понимать?**  
  `IHost` — это не просто контейнер зависимостей, а механизм, который задаёт правила запуска, работы и завершения приложения. В production это особенно важно для предсказуемого старта, завершения и наблюдаемости.

- **Экосистема и альтернативы**

| Компонент                        | Назначение                                              | Статус в задаче                                           |
| -------------------------------- | ------------------------------------------------------- | --------------------------------------------------------- |
| `Host.CreateDefaultBuilder()`    | Универсальный хост с defaults (значениями по умолчанию) | ✅ Используется                                           |
| `Host.CreateEmptyBuilder()`      | Минимальный хост без предустановок                      | ❌ Не рассматривается, требует ручной настройки           |
| `WebApplication.CreateBuilder()` | Хост для HTTP/Web API приложений                        | ❌ Не рассматривается, заточен под Kestrel/MVC            |
| `BackgroundService`              | Абстракция для долгоживущих фоновых задач               | ❌ Не рассматривается, расширяет `IHost`, не заменяет его |

- **Типичные ошибки**
  - Регистрируют сервисы хаотично, без структуры.
  - Смешивают bootstrap-логику с бизнес-логикой.
  - Не выносят конфигурацию в отдельные слои или методы.
  - Путают `IHost` с web-host-архитектурой.
  - Не учитывают порядок инициализации hosted services.

- **Production-практика**
  - Использовать `Host.CreateDefaultBuilder()` как стандартный путь для консольных и worker-приложений.
  - Все инфраструктурные зависимости регистрировать централизованно.
  - Не превращать `Program.cs` в место для сложной бизнес-логики.
  - Держать конфигурацию, логирование и DI в одном согласованном контуре.

## Секция 3: Управление жизненным циклом хоста

📚 **Теоретическая справка**

- **Что это?**  
  `RunAsync()` — асинхронный метод, запускающий хост и блокирующий выполнение текущего потока до получения сигнала завершения приложения.

- **Зачем это?**  
  Обеспечивает непрерывную работу консольного или фонового процесса, гарантирует корректное освобождение ресурсов (`graceful shutdown` — корректное завершение), предотвращает утечки подключений и прерывание фоновых операций.

- **Когда это уместно?**  
  Подходит для любых приложений, где важны управляемый запуск и корректная остановка: workers, фоновые процессы, сервисы, long-running консольные приложения.

- **Как это реализовано?**  
  Метод подписывается на системные события завершения (`Console.CancelKeyPress`, `SIGINT`/`SIGTERM`). При получении сигнала активируется `CancellationTokenSource`, который последовательно оповещает все зарегистрированные `IHostedService`. Остановка происходит в обратном порядке регистрации. После завершения всех фоновых задач метод снимает блокировку и возвращает управление коду, расположенному ниже в потоке выполнения.

- **Что важно понимать?**  
  Корректное завершение — часть архитектуры приложения, а не “дополнительная опция”. Если не учитывать shutdown-path, приложение может терять данные, оставлять незавершённые операции и вести себя нестабильно при остановке.

- **Экосистема и альтернативы**

| Метод управления                          | Поведение                          | Отличия от задачи                                                         |
| ----------------------------------------- | ---------------------------------- | ------------------------------------------------------------------------- |
| `RunAsync()`                              | Асинхронная блокировка до сигнала  | ✅ Используется                                                           |
| `Run()`                                   | Синхронная блокировка              | ❌ Не рекомендуется, может мешать async-flow                              |
| `StartAsync()` + `WaitForShutdownAsync()` | Раздельный запуск и ожидание       | ❌ Требует ручного управления токеном завершения                          |
| `StopAsync(TimeSpan)`                     | Программное завершение с таймаутом | ❌ Применяется для принудительной остановки, не для обработки сигналов OS |

- **Типичные ошибки**
  - Игнорируют сигналы завершения.
  - Не используют `CancellationToken`.
  - Не освобождают ресурсы при остановке.
  - Пишут фоновые задачи, которые невозможно корректно завершить.
  - Смешивают запуск приложения и рабочую логику в одном потоке без управления жизненным циклом.

- **Production-практика**
  - Всегда проектировать корректный shutdown-path.
  - Использовать `CancellationToken` во всех долгих операциях.
  - Проверять, что фоновые сервисы корректно завершаются.
  - Тестировать остановку приложения так же, как и запуск.

## Секция 4: Финализация базовой структуры

📚 **Теоретическая справка**

- **Что это?**  
  Архитектурный шаблон, объединяющий современные синтаксические возможности C# и стандартный паттерн инициализации хоста в компактный, поддерживаемый файл точки входа.

- **Зачем это?**  
  Формирует единый стандарт запуска для микросервисов и консольных утилит в .NET. Минимизация объёма кода снижает когнитивную нагрузку, упрощает code review (ревью кода) и ускоряет onboarding (онбординг, ввод в проект) новых разработчиков.

- **Когда это уместно?**  
  Подходит для небольших сервисов, консольных приложений, worker-проектов и вспомогательных утилит, где важна ясная и короткая точка входа.

- **Как это реализовано?**  
  На этапе компиляции цепочка вызовов разворачивается в единый асинхронный метод `Main`. JIT-компилятор применяет оптимизации к путям инициализации, а инфраструктурные сервисы загружаются лениво или при первом обращении. Компактность исходного кода не влияет на runtime-производительность, если архитектура регистрации и запуска остаётся чистой.

- **Что важно понимать?**  
  Упрощённый вход в приложение не означает упрощённую архитектуру вообще. Внутренние слои, сервисы и конфигурация всё равно должны быть отделены и оформлены явно.

- **Экосистема и альтернативы**

| Архитектура точки входа                                 | Сложность | Область применения                                          |
| ------------------------------------------------------- | --------- | ----------------------------------------------------------- |
| Минимальный `Program.cs` (<10 строк)                    | Низкая    | ✅ Используется в консолях, воркерах, микросервисах         |
| `Program.cs` + `Startup.cs`                             | Средняя   | ❌ Унаследовано из .NET Core 2.x/3.x, избыточно для .NET 6+ |
| Explicit factory pattern (явный фабричный паттерн)      | Высокая   | ❌ Применяется при сложной условной инициализации DI        |
| Composition root (корень композиции) в отдельной сборке | Высокая   | ❌ Используется в крупно-модульных enterprise-системах      |

- **Типичные ошибки**
  - Перегружают `Program.cs` логикой приложения.
  - Дублируют регистрацию зависимостей в нескольких местах.
  - Делают входной файл нечитаемым.
  - Не разделяют обязанности между bootstrap и domain layer (слоем домена).
  - Путают минимализм кода с отсутствием архитектуры.

- **Production-практика**
  - Держать точку входа короткой и прозрачной.
  - Выносить сложную регистрацию в отдельные методы или классы.
  - Поддерживать единый стиль запуска во всех сервисах проекта.
  - Оставлять в `Program.cs` только orchestration (оркестрацию) старта приложения.

## Секция 5: Диагностика сборки и зависимостей

📚 **Теоретическая справка**

- **Что это?**  
  Набор команд .NET CLI для инспекции состояния инструментария, трассировки процесса компиляции и анализа графа NuGet-зависимостей проекта.

- **Зачем это?**  
  Обеспечивает детерминированность сборок, ускоряет диагностику конфликтов версий, позволяет аудировать лицензионную чистоту и транзитивные зависимости, что критически важно для CI/CD (непрерывной интеграции и доставки).

- **Когда это уместно?**  
  Особенно полезно в командах, где важны воспроизводимые сборки, контроль зависимостей, audit trail (журнал аудита) и стабильность pipeline (конвейера) в CI.

- **Как это реализовано?**  
  `dotnet --info` считывает метаданные установленных SDK и рантаймов из системных реестров и переменных окружения. `dotnet build -v:d` запускает MSBuild с расширенным логгером, фиксируя порядок выполнения задач, время компиляции и разрешения типов. `dotnet list package` парсит `.csproj` и сгенерированный `obj/project.assets.json`, строя граф зависимостей с учётом политик версионирования и источников фидов.

- **Что важно понимать?**  
  Диагностика — это не разовая проверка, а часть управляемого процесса разработки. Если не отслеживать SDK, пакеты и сборочную цепочку, проект постепенно теряет воспроизводимость.

- **Экосистема и альтернативы**

| Инструмент                                      | Функционал                  | Отличия от задачи                                        |
| ----------------------------------------------- | --------------------------- | -------------------------------------------------------- |
| `dotnet --info` / `build -v:d` / `list package` | Стандартная CLI-диагностика | ✅ Используется                                          |
| `dotnet nuget locals`                           | Управление кэшем пакетов    | ❌ Не анализирует граф зависимостей проекта              |
| `dotnet-outdated-tool`                          | Поиск обновлений NuGet      | ❌ Внешний глобальный инструмент, не входит в SDK        |
| `MSBuildStructuredLog`                          | Визуализация графа сборки   | ❌ Требует генерации `.binlog` и стороннего просмотрщика |

- **Типичные ошибки**
  - Не проверяют версии SDK и рантайма.
  - Игнорируют транзитивные зависимости.
  - Не отслеживают дублирующиеся или устаревшие пакеты.
  - Диагностируют сборку только при падении CI.
  - Не сохраняют сборочные артефакты для анализа.

- **Production-практика**
  - Проверять `dotnet --info` перед стартом работ и в CI.
  - Контролировать список пакетов и версии зависимостей.
  - Использовать диагностику сборки как часть стандартного workflow.
  - Фиксировать изменения графа зависимостей при каждом обновлении.

## Секция 6: Документация и сдача модуля

📚 **Теоретическая справка**

- **Что это?**  
  Формализация инженерных решений в технической документации и стандартизированный процесс внесения изменений через изолированную ветку и механизм Pull Request (запрос на слияние).

- **Зачем это?**  
  Гарантирует прозрачность разработки, позволяет проводить экспертную оценку кода до интеграции, формирует аудиторский след для соответствия стандартам качества и упрощает поддержку проекта в долгосрочной перспективе.

- **Когда это уместно?**  
  Всегда, когда изменения затрагивают архитектуру, точки входа, инфраструктуру, сборку или поведение приложения. Особенно важно для командной разработки и CI/CD.

- **Как это реализовано?**  
  Git вычисляет diff (разницу) между снимками состояний веток, изолируя изменения. PR создаёт сущность в системе контроля версий, привязывает коммиты, инициирует автоматизированные проверки (CI), агрегирует комментарии и апрувы. Семантические префиксы в сообщениях коммитов (`feat:`, `chore:`, `fix:`) используются инструментами автоматической генерации CHANGELOG (журнала изменений) и классификации релизов.

- **Что важно понимать?**  
  Документация и процесс сдачи — это часть инженерного качества, а не формальность. Без них изменения теряют объяснимость, а команда — понимание, почему решение было принято именно так.

- **Экосистема и альтернативы**

| Практика                                         | Механизм                                             | Отличия от задачи                                           |
| ------------------------------------------------ | ---------------------------------------------------- | ----------------------------------------------------------- |
| Pull Request / Merge Request (запрос на слияние) | Изолированное ревью, CI-проверки, история обсуждений | ✅ Используется                                             |
| Trunk-based development (разработка через trunk) | Прямые коммиты в `main`, feature flags               | ❌ Требует высокой дисциплины и автоматических тестов       |
| Squash Merge (сведение коммитов в один)          | Объединение всех коммитов перед мёржем               | ❌ Упрощает историю, но теряет детализацию шагов разработки |
| Email/Issue-ревью                                | Внесение изменений без Git-механики ревью            | ❌ Не масштабируется, отсутствует автоматизация и аудит     |

- **Типичные ошибки**
  - Делают PR без описания архитектурного смысла изменений.
  - Не прикладывают проверяемые результаты.
  - Пишут коммиты без понятных префиксов.
  - Сливают изменения без CI-проверок.
  - Не фиксируют важные решения в документации.

- **Production-практика**
  - Любое изменение точки входа должно сопровождаться описанием.
  - Коммиты должны быть понятными и пригодными для автоматизации.
  - PR должен отвечать на вопрос: что поменялось, почему и как это проверено.
  - При архитектурных изменениях желательно оставлять краткий ADR (Architecture Decision Record — запись архитектурного решения).

## Итог

`Console Modernization (модернизация консольного приложения)` в .NET 10 — это не просто сокращение `Program.cs`, а переход к управляемой, воспроизводимой и поддерживаемой точке входа. Главная цель — сделать запуск приложения прозрачным, lifecycle (жизненный цикл) — корректным, а сопровождение — предсказуемым.
