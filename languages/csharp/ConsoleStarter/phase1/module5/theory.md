# Модуль 5 — BackgroundService & Cancellation

> Цель: Реализовать фоновый воркер с корректной отменой через CancellationToken.

---

## Вступление

Этот модуль вводит долгоживущие фоновые процессы в .NET 10 и показывает, как правильно завершать их без потери данных и без принудительного убийства процесса. Здесь важно понять не только, как запустить воркер через `BackgroundService`, но и как связать его работу с `CancellationToken`, `IHostApplicationLifetime` и корректным graceful shutdown.

Модуль 5 продолжает логику предыдущих модулей: после конфигурации, DI и runtime reconfiguration курс переходит к устойчивому фоновому выполнению. Это естественный шаг от “настроить приложение” к “заставить его работать долго и завершаться безопасно”.

---

## Секция 1: Базовая реализация BackgroundService

**В этой секции рассматривается стандартный способ создания фонового воркера в .NET 10 через `BackgroundService`. Он даёт готовую инфраструктуру для долгоживущих задач и оставляет разработчику только `ExecuteAsync`.**

📚 **Теоретическая справка**

- **Что это?**  
   `BackgroundService` — базовый класс для фоновых задач, который реализует контракт `IHostedService` и предназначен для долгоживущих процессов.
- **Зачем это?**  
   Он снимает с разработчика рутину по управлению запуском и остановкой сервиса и оставляет только бизнес-логику внутри `ExecuteAsync`.
- **Как это реализовано?**  
   Хост вызывает `StartAsync`, внутри которого `BackgroundService` подготавливает выполнение `ExecuteAsync(CancellationToken)` в фоновом режиме. Затем хост удерживает воркер активным до остановки приложения. Регистрация через `AddHostedService<T>()` добавляет сервис в контейнер и подключает его к жизненному циклу приложения.learn.microsoft+1
- **Что важно понимать?**  
   `BackgroundService` — это не отдельный процесс и не поток вручную, а управляемая часть Generic Host. Это делает его удобным для консольных приложений, worker-сервисов и инфраструктурных задач.
- **Экосистема и альтернативы**

| Подход к фоновым задачам           | Уровень абстракции      | Область применения                                             |
| ---------------------------------- | ----------------------- | -------------------------------------------------------------- |
| `BackgroundService`                | Высокий, готовый шаблон | ✅ Консольные воркеры, демоны, периодическая обработка         |
| Прямая реализация `IHostedService` | Средний                 | ❌ Требует ручной обработки `StartAsync` и `StopAsync`         |
| `Thread` / `Timer`                 | Низкий                  | ❌ Legacy-подход, нет интеграции с DI и жизненным циклом хоста |
| Сторонние планировщики             | Очень высокий           | ❌ Очереди, cron, распределённая инфраструктура                |

- **Типичные ошибки**
  - Переносят бизнес-логику в `Program.cs`.
  - Делают бесконечный цикл без контроля отмены.
  - Используют `BackgroundService` как замену обычному методу.
  - Пытаются вручную управлять потоком вместо хоста.
  - Не регистрируют сервис через `AddHostedService`.
- **Production-практика**
  - Держать воркер изолированным в отдельном классе.
  - Внутри `ExecuteAsync` выполнять только задачу воркера.
  - Использовать хост для запуска и остановки сервиса.
  - Не смешивать startup-логику и рабочий цикл.
  - Делать воркер предсказуемым и коротким по структуре.

## Секция 2: Кооперативная отмена через CancellationToken

**Дальше важно научить воркер завершаться не насильственно, а по сигналу отмены. В .NET 10 это делается через кооперативную отмену, где сам код проверяет `CancellationToken` и выходит из цикла.**

📚 **Теоретическая справка**

- **Что это?**  
   Кооперативная отмена — это модель, при которой задача сама отслеживает `CancellationToken` и завершает работу в безопасной точке.
- **Зачем это?**  
   Она позволяет завершать работу без повреждения данных, утечек ресурсов и неожиданных обрывов I/O-операций.
- **Как это реализовано?**  
   `CancellationTokenSource` активирует токен при остановке приложения. Воркер проверяет `stoppingToken.IsCancellationRequested` в цикле, а асинхронные операции, такие как `Task.Delay`, получают токен и завершаются через `OperationCanceledException`.learn.microsoft+1
- **Что важно понимать?**  
   Отмена должна быть частью дизайна воркера, а не дополнительной обработкой после ошибки. Если задача может долго выполняться, она должна регулярно проверять токен.
- **Экосистема и альтернативы**

| Механизм завершения | Детерминизм | Безопасность состояния                |
| ------------------- | ----------- | ------------------------------------- |
| `CancellationToken` | Высокий     | ✅ Гарантирует управляемое завершение |
| `Thread.Abort()`    | Низкий      | ❌ Небезопасно и устарело             |
| `Process.Kill`      | Нулевой     | ❌ Мгновенное завершение без очистки  |
| Таймаут без токена  | Средний     | ❌ Не управляет логикой завершения    |

- **Типичные ошибки**
  - Не передают токен в `Task.Delay` и I/O-операции.
  - Проверяют отмену слишком редко.
  - Путают `OperationCanceledException` с ошибкой выполнения.
  - Завершают воркер без логики финальной очистки.
  - Надеются на “автоматическое” завершение без своей проверки токена.
- **Production-практика**
  - Проверять токен в каждой итерации цикла.
  - Передавать токен во все длительные async-операции.
  - Обрабатывать отмену отдельно от ошибок.
  - Делать завершение мягким и предсказуемым.
  - Оставлять код чтения токена простым и очевидным.

## Секция 3: Graceful shutdown и очистка ресурсов

**После отмены воркер должен не просто остановиться, а корректно освободить ресурсы. Это особенно важно для файлов, соединений, очередей и любой внешней инфраструктуры.**

📚 **Теоретическая справка**

- **Что это?**  
   Graceful shutdown — это контролируемое завершение работы, при котором сервис дорабатывает текущую операцию и освобождает ресурсы.
- **Зачем это?**  
   Такой подход предотвращает потерю данных и делает остановку воркера безопасной для внешних систем.
- **Как это реализовано?**  
   Основной цикл оборачивается в `try/finally` или `try/catch/finally`. В `finally` выполняется очистка: закрытие соединений, освобождение файловых дескрипторов, финальное логирование. Если сервис использует `IDisposable` или `IAsyncDisposable`, контейнер DI также вызывает очистку после завершения хоста.[learn.microsoft](https://learn.microsoft.com/en-us/aspnet/core/fundamentals/host/hosted-services?view=aspnetcore-10.0)
- **Что важно понимать?**  
   Завершение процесса и завершение воркера — не одно и то же. Сначала задача должна закончить работу, и только потом хост может закрыть приложение.
- **Экосистема и альтернативы**

| Паттерн очистки              | Гарантия выполнения | Область применения                    |
| ---------------------------- | ------------------- | ------------------------------------- |
| `try/finally`                | Высокая             | ✅ Асинхронные циклы и фоновые задачи |
| `using` / `await using`      | Высокая             | ✅ Локальные ресурсы и соединения     |
| Очистка только в `StopAsync` | Средняя             | ❌ Легко пропустить при исключении    |
| Finalizer                    | Низкая              | ❌ Только для неуправляемых ресурсов  |

- **Типичные ошибки**
  - Не закрывают соединения при остановке.
  - Дублируют логику очистки в нескольких местах.
  - Полагаются только на финализаторы.
  - Убеждены, что `StopAsync` всегда решает вопрос очистки.
  - Не логируют финальное завершение воркера.
- **Production-практика**
  - Использовать `finally` для гарантированной очистки.
  - Освобождать внешние ресурсы сразу после завершения работы.
  - Делать shutdown-путь симметричным startup-пути.
  - Проверять корректность завершения в реальных сценариях остановки.
  - Оставлять финальный код коротким и прозрачным.

## Секция 4: Интеграционный тест корректной отмены

**После реализации воркера важно проверить, что он действительно завершается быстро и безопасно. Это подтверждает, что код готов к работе в контейнерах и оркестраторах.**

📚 **Теоретическая справка**

- **Что это?**  
   Интеграционная проверка времени завершения и корректности остановки фонового сервиса.
- **Зачем это?**  
   Она показывает, что приложение успевает завершиться до принудительного shutdown со стороны среды выполнения или оркестратора.
- **Как это реализовано?**  
   Воркер запускается, затем получает сигнал завершения, после чего хост ждёт окончания `ExecuteAsync` и только потом закрывается. Все операции должны быть асинхронными и принимать токен отмены.learn.microsoft+1
- **Что важно понимать?**  
   Тест на завершение — это не формальность. Он проверяет, что воркер не зависает и не требует принудительного убийства процесса.
- **Экосистема и альтернативы**

| Стратегия завершения        | Влияние на оркестратор          | Сложность реализации              |
| --------------------------- | ------------------------------- | --------------------------------- |
| Graceful shutdown           | Позитивное                      | Низкая                            |
| Долгая блокирующая операция | Риск принудительного завершения | Высокая                           |
| Игнорирование токена        | Критическое                     | Низкая по коду, высокая по рискам |
| Hard kill                   | Разрушительное                  | Не подходит для курса             |

- **Типичные ошибки**
  - Добавляют блокирующие вызовы в `ExecuteAsync`.
  - Не измеряют время завершения.
  - Проверяют только запуск, но не остановку.
  - Не замечают утечки ресурсов после отмены.
  - Считают, что Ctrl+C автоматически гарантирует корректное завершение.
- **Production-практика**
  - Проверять как запуск, так и остановку воркера.
  - Держать завершение коротким и асинхронным.
  - Не выполнять тяжёлую работу в пути shutdown.
  - Логировать этапы остановки.
  - Тестировать поведение на реальном сигнале отмены.

## Секция 5: События жизненного цикла через IHostApplicationLifetime

**Чтобы управлять не только самим воркером, но и общими фазами жизни приложения, используется `IHostApplicationLifetime`. Это удобно для логирования, метрик и действий на старте и остановке.**

📚 **Теоретическая справка**

- **Что это?**  
   `IHostApplicationLifetime` — сервис для подписки на события жизненного цикла приложения: запуск, остановка и завершение.
- **Зачем это?**  
   Он позволяет выполнять общие инфраструктурные действия независимо от конкретного `BackgroundService`.
- **Как это реализовано?**  
   Через `ApplicationStopping.Register(...)` можно добавить callback, который выполнится при начале остановки хоста. Это полезно для финального логирования, отправки метрик и координации внешних систем.[learn.microsoft](https://learn.microsoft.com/en-us/aspnet/core/fundamentals/host/hosted-services?view=aspnetcore-10.0)
- **Что важно понимать?**  
   Этот механизм дополняет `BackgroundService`, но не заменяет его. Воркеры отвечают за работу, lifetime-события — за orchestration.
- **Экосистема и альтернативы**

| Механизм отслеживания состояния | Контекст вызова               | Область применения                             |
| ------------------------------- | ----------------------------- | ---------------------------------------------- |
| `IHostApplicationLifetime`      | Глобальный                    | ✅ Инфраструктурные хуки, метрики, логирование |
| `StopAsync`                     | На уровне конкретного сервиса | ❌ Локальная логика остановки                  |
| `ProcessExit`                   | Низкоуровневый                | ❌ Legacy и слабая интеграция с DI             |
| OS signals напрямую             | Системный                     | ❌ Слишком низкий уровень для курса            |

- **Типичные ошибки**
  - Делают тяжёлую работу внутри `ApplicationStopping`.
  - Смешивают lifecycle-события с бизнес-логикой воркера.
  - Не понимают разницу между stop-событиями и самим `ExecuteAsync`.
  - Используют `IHostApplicationLifetime` как замену нормальной отмене.
  - Логируют слишком поздно, когда хост уже почти остановлен.
- **Production-практика**
  - Использовать события lifetime для коротких инфраструктурных действий.
  - Не перегружать callback тяжёлой работой.
  - Логировать фазу остановки до закрытия ресурсов.
  - Сочетать lifetime-события с токенами отмены.
  - Держать orchestration отдельно от рабочего цикла.

## Секция 6: Документация и сдача модуля

**Когда воркер и отмена уже работают, нужно зафиксировать архитектуру, сценарии проверки и ожидаемое поведение в документации. Это завершает модуль и делает его воспроизводимым.**

📚 **Теоретическая справка**

- **Что это?**  
   Документирование фонового сервиса, его остановки и критериев проверки корректной отмены.
- **Зачем это?**  
   Это помогает воспроизводить поведение, проверять shutdown и передавать модуль другому разработчику без потери контекста.
- **Как это реализовано?**  
   В README фиксируются структура воркера, способ проверки Ctrl+C, поведение токена отмены и ожидаемые логи. Изменения оформляются отдельной веткой и проходят PR-ревью.
- **Что важно понимать?**  
   Для фоновых процессов особенно важно документировать не только запуск, но и остановку.
- **Экосистема и альтернативы**

| Практика сдачи | Механизм | Отличия от задачи |
| -------------- | -------- | ----------------- |

| Практика сдачи                        | Механизм                       | Отличия от задачи        |
| ------------------------------------- | ------------------------------ | ------------------------ |
| PR с инструкцией и проверкой shutdown | Изолированное ревью, CI, аудит | ✅ Используется          |
| Direct commit в main                  | Быстро, но рискованно          | ❌ Без контроля качества |
| Squash merge                          | Упрощает историю               | ❌ Теряется поэтапность  |
| Только релиз-ноты                     | Недостаточно для проверки      | ❌ Нет привязки к коду   |

- **Типичные ошибки**
  - Не описывают, как проверять отмену.
  - Не фиксируют ожидаемую последовательность логов.
  - Считают документацию второстепенной.
  - Прячут shutdown-поведение внутри кода без объяснения.
  - Не указывают критерии успешной остановки.
- **Production-практика**
  - Документировать сценарий запуска и остановки.
  - Писать, как проверить корректный shutdown.
  - Описывать, какие ресурсы освобождаются.
  - Сохранять модуль в виде воспроизводимого шаблона.
  - Держать README коротким, но проверяемым.
