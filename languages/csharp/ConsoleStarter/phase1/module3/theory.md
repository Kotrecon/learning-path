# Модуль 3 — Configuration Pipeline

> Цель: Настроить конфигурацию с приоритетами: CLI > ENV > JSON.

---

## Секция 1: Базовый провайдер appsettings.json

### Задача 3.1: Настроить чтение конфигурации из appsettings.json

**Описание:** Создать файл appsettings.json с иерархическими настройками. Подключить IConfiguration и прочитать значение через индексатор с разделителем ":". Убедиться, что изменение файла требует перезапуска приложения.

**Способы достижения:**

- Создать `appsettings.json`: `{ "App": { "Name": "MyApp", "Timeout": 30 } }`
- В Program.cs получить конфиг: `var config = host.Services.GetRequiredService<IConfiguration>()`
- Вывести значение: `Console.WriteLine(config["App:Name"])` и проверить вывод "MyApp"
- Закоммитить: `git add .`
  `git commit -m "feat(module3): add appsettings.json provider"`

**Результат:** Файл appsettings.json, вывод консоли со значением из конфига, коммит с настройкой JSON-провайдера в репозитории.

📚 **Теоретическая справка**

- **Что это?** `JsonConfigurationProvider` — компонент инфраструктуры, парсящий JSON-файлы в плоское дерево ключей-значений, доступное через интерфейс `IConfiguration`.
- **Зачем это?** Отделяет конфигурацию от бинарного кода, позволяет версионировать настройки через Git, поддерживает иерархическую структуру данных и удобен для командной работы и деплоя.
- **Как это реализовано?** `CreateApplicationBuilder` автоматически регистрирует провайдер в начале цепочки. При инициализации файл читается, JSON десериализуется во вложенные объекты. Индексатор `IConfiguration["App:Name"]` рекурсивно обходит дерево, используя `:` как разделитель сегментов. По умолчанию провайдер загружает файл один раз при старте; изменения не отслеживаются до явной настройки `reloadOnChange`.
- **Экосистема и альтернативы**

| Источник конфигурации            | Формат данных                       | Область применения                                       |
| -------------------------------- | ----------------------------------- | -------------------------------------------------------- |
| `appsettings.json`               | Иерархический JSON                  | ✅ Базовая конфигурация, версия в VCS                    |
| `appsettings.{Environment}.json` | JSON с оверлеем по среде            | ❌ Переопределение для Dev/Stage/Prod                    |
| `XML / INI провайдеры`           | Структурированные текстовые форматы | ❌ Наследие .NET Framework, нишевые интеграции           |
| Embedded Resources               | Вшитые в сборку файлы               | ❌ Статичные конфигурации, не подлежат runtime-изменению |

## Секция 2: Переопределение через переменные окружения

### Задача 3.2: Реализовать приоритет ENV-переменных над JSON

**Описание:** Научиться использовать переменные окружения с разделителем "\_\_" для переопределения вложенных ключей конфигурации. Проверить, что значение из ENV имеет приоритет над JSON без изменения кода.

**Способы достижения:**

- В PowerShell установить: `$env:App__Timeout="120"`
- Запустить `dotnet run` и убедиться, что выводится "120", а не "30" из JSON
- Проверить порядок провайдеров через `config.GetChildren()` или отладчик
- Закоммитить: `git commit -am "feat(module3): add Environment Variables provider"`

**Результат:** Вывод приложения с переопределённым значением из ENV, подтверждение приоритета провайдеров, коммит в истории репозитория.

📚 **Теоретическая справка**

- **Что это?** `EnvironmentVariablesConfigurationProvider` — механизм инжекции конфигурационных значений из контекста выполнения процесса (ОС, Docker, CI/CD).
- **Зачем это?** Обеспечивает безопасность (секреты не хранятся в репозитории), поддерживает 12-factor app принципы, позволяет динамически менять поведение приложения в разных средах без пересборки.
- **Как это реализовано?** Провайдер считывает `Environment.GetEnvironmentVariables()`. Поскольку ОС не поддерживает вложенные ключи, .NET использует двойное подчеркивание `__` как escape-символ, транслируемый в `:` во время загрузки. Провайдер регистрируется в цепочке после JSON. При резолве `IConfiguration` итерирует провайдеры в порядке добавления; при совпадении ключа значение из последнего добавленного провайдера перезаписывает предыдущее.
- **Экосистема и альтернативы**

| Механизм ENV-маппинга        | Синтаксис                 | Область применения                                  |
| ---------------------------- | ------------------------- | --------------------------------------------------- |
| `__` (двойное подчеркивание) | `App__Timeout=120`        | ✅ Кроссплатформенный стандарт .NET                 |
| Префиксная фильтрация        | `DOTNET_` или `MYAPP_`    | ❌ Изоляция переменных приложения от системных      |
| User Secrets                 | `dotnet user-secrets set` | ❌ Локальная разработка, не для контейнеров/CI      |
| `:` (прямой)                 | `App:Timeout=120`         | ❌ Неработоспособен в Windows ENV (запрет символов) |

## Секция 3: Наивысший приоритет аргументов командной строки

### Задача 3.3: Верифицировать приоритет CLI-аргументов над ENV и JSON

**Описание:** Понять, что аргументы командной строки имеют наивысший приоритет в цепочке провайдеров. Протестировать комбинации значений в трёх источниках и зафиксировать итоговое поведение.

**Способы достижения:**

- Задать значение в JSON (30), ENV (120) и запустить с `--App:Timeout=300`
- Убедиться, что приложение выводит "300" — значение из CLI
- Создать таблицу тестовых комбинаций в `audit/cli-precedence-test.md`
- Закоммитить: `git commit -am "feat(module3): verify CLI argument priority"`

**Результат:** Таблица результатов тестов приоритета, подтверждение победы CLI-значения, коммит с верификацией в репозитории.

📚 **Теоретическая справка**

- **Что это?** `CommandLineConfigurationProvider` — парсер аргументов запуска, преобразующий флаги командной строки в конфигурационные ключи.
- **Зачем это?** Позволяет операторам и оркестраторам переопределять настройки на лету, упрощает ручное тестирование и отладку, обеспечивает гибкость запуска без модификации файлов или ENV.
- **Как это реализовано?** Аргументы передаются в `CreateDefaultBuilder(args)`. Провайдер добавляется в конец коллекции `ConfigurationProviders`. Парсер распознает префиксы `--`, `/`, `-` и ключи в форматах `--Key=Value`, `--Key Value` или `--Key:Value`. При разрешении конфигурации движок `ConfigurationRoot` проходит провайдеры от первого к последнему; значение из CLI, будучи в конце цепочки, всегда выигрывает при конфликте ключей.
- **Экосистема и альтернативы**

| Формат аргументов                       | Синтаксис           | Область применения                                |
| --------------------------------------- | ------------------- | ------------------------------------------------- |
| `--Key=Value`                           | `--App:Timeout=300` | ✅ Современный стандарт .NET                      |
| `--Key Value`                           | `--App:Timeout 300` | ❌ Альтернативный синтаксис, менее явный          |
| `/Key=Value`                            | `/App:Timeout=300`  | ❌ Legacy Windows-совместимость                   |
| `ConfigurationBuilder.AddCommandLine()` | Ручная регистрация  | ❌ Требуется при отказе от `CreateDefaultBuilder` |

## Секция 4: Автоматизация проверки приоритетов

### Задача 3.4: Создать скрипт для автоматической проверки цепочки приоритетов

**Описание:** Написать PowerShell-скрипт, который устанавливает ENV, запускает приложение с аргументами и сравнивает вывод с ожидаемым значением. Автоматизировать регрессионное тестирование конфигурации.

**Способы достижения:**

- Создать `test-config-precedence.ps1`: установить ENV, запустить с `--args`, сравнить вывод
- Добавить проверки для всех комбинаций: только JSON, JSON+ENV, JSON+ENV+CLI
- Вернуть код выхода 0 при успешном прохождении всех тестов
- Закоммитить скрипт: `git add .`
  `git commit -m "feat(module3): automate precedence verification"`

**Результат:** Рабочий скрипт test-config-precedence.ps1, вывод "All precedence tests passed", коммит с автоматизацией в репозитории.

📚 **Теоретическая справка**

- **Что это?** Инфраструктурный скрипт, оркестрирующий запуск процесса, инъекцию переменных окружения и валидацию `stdout` для подтверждения корректности конфигурационного пайплайна.
- **Зачем это?** Гарантирует детерминизм поведения при миграциях SDK, интегрируется в CI/CD конвейеры, исключает человеческий фактор при ручных проверках и фиксирует контракт приоритетов.
- **Как это реализовано?** Скрипт использует хуки оболочки для изоляции ENV (`$env:VAR`, очистка после теста), вызывает `dotnet run` с перехватом вывода, применяет строковое сравнение или regex-матчинг для валидации ожидаемого результата. Возврат `exit 0`/`exit 1` стандартизирует интеграцию с системами сборки. Полагается на стабильность порядка инициализации `ConfigurationBuilder`.
- **Экосистема и альтернативы**

| Метод валидации                                | Уровень изоляции         | Сложность интеграции                              |
| ---------------------------------------------- | ------------------------ | ------------------------------------------------- |
| Оболочечный скрипт (PS/Bash)                   | Высокий (полный процесс) | ✅ Низкая, нативный CI-интеграция                 |
| `xUnit` + `TestServer`/`WebApplicationFactory` | Высокий (в памяти)       | ❌ Требует мок-хоста, избыточно для консоли       |
| `dotnet run` + ручной diff                     | Низкий                   | ❌ Не масштабируется, подвержен ошибкам           |
| Contract-тесты (Pact)                          | Средний                  | ❌ Сфокусировано на API-контрактах, не на конфиге |

## Секция 5: Типизированный доступ через IOptions<T>

### Задача 3.5: Настроить биндинг конфигурации в типизированный объект

**Описание:** Создать record AppSettings с свойствами Name и Timeout. Настроить биндинг секции "App" через Configure<T>. Резолвить IOptions<AppSettings> и вывести типизированные значения.

**Способы достижения:**

- Создать `AppSettings.cs`: `public record AppSettings { public string Name { get; init; } public int Timeout { get; init; } }`
- В Program.cs добавить: `builder.Services.Configure<AppSettings>(builder.Configuration.GetSection("App"))`
- Резолвить и вывести: `var settings = host.Services.GetRequiredService<IOptions<AppSettings>>().Value`
- Закоммитить: `git commit -am "feat(module3): add IOptions<T> binding"`

**Результат:** Класс AppSettings.cs, успешный биндинг значений из конфига, коммит с типизированным доступом в репозитории.

- **Важная оговорка**: `IOptions<T>.Value` кэшируется при **первом запросе**, а не при старте хоста. Изменения в `appsettings.json` после первого резолва не подхватываются. Для hot-reload нужен `IOptionsMonitor<T>`.

📚 **Теоретическая справка**

- **Что это?** Паттерн `Options<T>`, обеспечивающий строго типизированный, валидируемый и удобный для DI доступ к конфигурационным секциям через `IConfigurationSection`.
- **Зачем это?** Исключает ошибки строковых ключей (`"App:Timeout"`), обеспечивает compile-time проверку типов, поддерживает валидацию ограничений на старте и интегрируется в контейнер зависимостей без ручного парсинга.
- **Как это реализовано?** `Configure<T>` регистрирует фабрику в `IServiceCollection`. При первом запросе `IOptions<T>.Value` `ConfigurationBinder` использует рефлекссию для сопоставления имен свойств POCO с ключами конфига, выполняет приведение типов и кэширует результат в виде синглтона. Последующие запросы возвращают кэшированный экземпляр без повторного парсинга.
- **Экосистема и альтернативы**

| Интерфейс доступа              | Стратегия обновления               | Область применения                                        |
| ------------------------------ | ---------------------------------- | --------------------------------------------------------- |
| `IOptions<T>`                  | Единожды при старте (синглтон)     | ✅ Консольные приложения, статичный конфиг                |
| `IOptionsSnapshot<T>`          | Перезагрузка на каждый скоуп       | ❌ Веб-запросы, где конфиг может меняться между запросами |
| `IOptionsMonitor<T>`           | Реактивные обновления (`OnChange`) | ❌ Горячая перезагрузка без рестарта, фоновые сервисы     |
| `ConfigurationBinder.Get<T>()` | Прямой вызов, без DI               | ❌ Ручное инстанцирование, не тестируется через моки      |

## Секция 6: Документация и сдача модуля

### Задача 3.6: Подготовить документацию и открыть PR для сдачи модуля

**Описание:** Создать README.md с примерами настройки каждого источника конфигурации и проверки приоритетов. Провести финальный прогон скрипта автоматизации. Открыть Pull Request для сдачи модуля.

**Способы достижения:**

- Создать `module3/README.md` с примерами: JSON, ENV, CLI, IOptions<T>
- Прогнать `test-config-precedence.ps1` и убедиться в прохождении всех тестов
- Создать ветку `feature/module3-complete`, запушить и открыть PR с чек-листом
- Приложить к PR таблицу приоритетов и примеры кода биндинга

**Результат:** Документированный модуль в README.md, открыт PR с меткой "Модуль 3 завершён", скрипт приоритетов проходит, все критерии подтверждены.

📚 **Теоретическая справка**

- **Что это?** Формализация конфигурационных контрактов в технической документации и стандартизированный процесс интеграции изменений через изолированную ветку и код-ревью.
- **Зачем это?** Фиксирует ожидаемое поведение пайплайна приоритетов для будущих разработчиков, обеспечивает аудиторский след конфигурационных изменений и гарантирует, что автоматические проверки прошли до слияния в основную ветку.
- **Как это реализовано?** Документация описывает контракты данных (форматы ключей, типы свойств, таблицы приоритетов). PR создает изолированный контекст ревью, запускает CI-пайплайн (включая скрипты валидации конфига), агрегирует аппрувы и сохраняет историю изменений. Семантические префиксы коммитов классифицируют изменения инфраструктуры для автоматической генерации релизных заметок.
- **Экосистема и альтернативы**

| Практика сдачи                | Механизм                                              | Отличия от задачи                                     |
| ----------------------------- | ----------------------------------------------------- | ----------------------------------------------------- |
| PR с документацией контрактов | Изолированное ревью, CI-валидация, исторический аудит | ✅ Используется                                       |
| Direct commit в main          | Мгновенный деплой без ревью                           | ❌ Высокий риск поломки конфигурационного пайплайна   |
| Squash merge                  | Атомарное объединение коммитов                        | ❌ Теряется пошаговая история настройки провайдеров   |
| Release-ноты без PR           | Ручное описание изменений                             | ❌ Отсутствие автоматической привязки к коду и тестам |

- Аргумент командной строки --App:Timeout=300 имеет наивысший приоритет

- Все изменения зафиксированы в Git с понятной историей коммитов
