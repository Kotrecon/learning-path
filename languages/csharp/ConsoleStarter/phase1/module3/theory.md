# Модуль 3 — Configuration Pipeline

> Цель: настроить конфигурацию с приоритетами: CLI > ENV > JSON.

## Вступление

Этот модуль показывает, как .NET собирает конфигурацию из нескольких источников и почему порядок их подключения важен. В консольном приложении особенно полезно понимать, откуда именно берётся итоговое значение: из `appsettings.json`, из переменной окружения или из аргумента командной строки.

---

## Секция 1: Базовый провайдер `appsettings.json`

**В этой секции рассматривается базовый источник конфигурации приложения — `appsettings.json`. Он хранит настройки в удобном JSON-формате, а система конфигурации .NET читает их через `IConfiguration`, превращая файл в набор доступных ключей и значений.**

📚 **Теоретическая справка**

- **Что это?**  
    `JsonConfigurationProvider` — это компонент, который читает JSON-конфигурацию и превращает её в набор ключей и значений, доступных через `IConfiguration`.

- **Зачем это?**  
    Такой подход отделяет настройки от кода, упрощает сопровождение, позволяет хранить конфигурацию в Git и удобно переопределять её для разных сред без правки исходников.

- **Как это реализовано?**  
    `CreateApplicationBuilder` автоматически подключает провайдер `appsettings.json` в начале цепочки источников конфигурации. При запуске файл читается, JSON преобразуется в дерево ключей и значений, а доступ к вложенным данным выполняется через двоеточие, например `App:Name`. По умолчанию файл загружается один раз при старте, а обновление во время работы возможно только при включённом `reloadOnChange`.

- **Что важно понимать?**  
    `appsettings.json` задаёт базовые значения, которые затем могут быть переопределены другими источниками. Его задача — дать приложению стабильную стартовую конфигурацию.

- **Экосистема и альтернативы**

| Источник конфигурации             | Формат данных                        | Область применения                              |
| --------------------------------- | ------------------------------------ | ----------------------------------------------- |
| `appsettings.json`                | Иерархический JSON                   | ✅ Базовая конфигурация, версия в VCS           |
| `appsettings.{Environment}.json`  | JSON с оверлеем по среде             | ✅ Переопределение для Dev/Stage/Prod           |
| `XML / INI`                       | Текстовые структурированные форматы  | ❌ Наследие старых решений, нишевые интеграции  |
| Embedded Resources                | Файлы внутри сборки                  | ❌ Статичные параметры без runtime-изменения    |

- **Типичные ошибки**
    - Не включают `appsettings.json` в проект (`CopyToOutputDirectory` не настроен).
    - Используют плоскую структуру JSON вместо иерархической, что усложняет группировку настроек.
    - Записывают секреты в `appsettings.json` вместо ENV или User Secrets.
    - Ожидают hot-reload без включения `reloadOnChange=true`.
    - Путают `appsettings.json` с `appsettings.{Environment}.json` и дублируют настройки.

- **Production-практика**
    - Хранить базовые настройки в `appsettings.json`, а средовые — в `appsettings.{Environment}.json`.
    - Никогда не коммитить секреты в JSON-файлы.
    - Использовать иерархическую структуру с ясными группами (например, `App:Timeout`, `App:Name`).
    - Включать `reloadOnChange=true` только там, где действительно нужно динамическое обновление.
    - Версионировать конфигурацию вместе с кодом и явно документировать изменения.

## Секция 2: Переопределение через переменные окружения

**Дальше конфигурация поднимается на уровень среды запуска: сначала базовые значения из JSON, а затем — переопределения из переменных окружения.**

📚 **Теоретическая справка**

- **Что это?**  
    `EnvironmentVariablesConfigurationProvider` — механизм получения конфигурационных значений из переменных окружения процесса.

- **Зачем это?**  
    Он позволяет выносить секреты и средовые настройки из репозитория, поддерживает Docker и CI/CD и даёт возможность менять поведение приложения без пересборки.

- **Как это реализовано?**  
    Провайдер читает `Environment.GetEnvironmentVariables()`. Поскольку в переменных окружения нельзя удобно использовать вложенные ключи, .NET применяет `__` как замену `:`. Например, `App__Timeout` превращается в `App:Timeout`. Если один и тот же ключ встречается в нескольких источниках, побеждает тот, который добавлен позже.

- **Что важно понимать?**  
    Переменные окружения удобны для настроек, зависящих от среды. Это хороший слой для секретов, адресов сервисов и параметров запуска, но не для бизнес-логики.

- **Экосистема и альтернативы**

| Механизм ENV-маппинга         | Синтаксис                  | Область применения                                   |
| ----------------------------- | -------------------------- | ---------------------------------------------------- |
| `__` (двойное подчеркивание)  | `App__Timeout=120`         | ✅ Кроссплатформенный стандарт .NET                  |
| Префиксная фильтрация         | `DOTNET_` или `MYAPP_`     | ✅ Изоляция переменных приложения от системных       |
| User Secrets                  | `dotnet user-secrets set`  | ✅ Локальная разработка, не для контейнеров/CI       |
| `:` (прямой)                  | `App:Timeout=120`          | ❌ Неработоспособен в Windows ENV (запрет символов)  |

- **Типичные ошибки**
    - Используют `:` вместо `__` в переменных окружения (не работает на Windows).
    - Хранят секреты в JSON вместо ENV.
    - Не фильтруют переменные по префиксу (`MYAPP_`), что приводит к конфликтам с системными переменными.
    - Ожидают, что ENV подхватится без перезапуска приложения.
    - Путают Case-sensitive поведение на Linux (APP_TIMEOUT ≠ App\_\_Timeout).

- **Production-практика**
    - Всегда использовать `__` для вложенных ключей в ENV.
    - Добавлять префикс ко всем переменным приложения (`MYAPP__Timeout`, `MYAPP__ConnectionString`).
    - Хранить секреты в ENV или специализированных хранилищах (Azure Key Vault, AWS Secrets Manager).
    - Использовать ENV для адресов сервисов, портов и путей в контейнерах.
    - Документировать все ENV-переменные в README с описанием назначения.

## Секция 3: Наивысший приоритет аргументов командной строки

**После ENV слой конфигурации добирается до самого локального уровня — параметров запуска. Именно здесь удобно точечно переопределять значения без изменения файлов и переменных окружения.**

📚 **Теоретическая справка**

- **Что это?**  
    `CommandLineConfigurationProvider` — парсер аргументов запуска, преобразующий флаги командной строки в конфигурационные ключи.

- **Зачем это?**  
    Он позволяет быстро переопределять настройки при запуске, удобно использовать приложение в тестах, локальной отладке и операционных сценариях.

- **Как это реализовано?**  
    Аргументы передаются в `CreateApplicationBuilder(args)`. Провайдер добавляется в цепочку конфигурации, и при разборе аргументов форматы вроде `--Key=Value` или `--Key Value` превращаются в пары ключ-значение. Поскольку CLI-источник подключается последним, он получает наивысший приоритет и перекрывает JSON и ENV.

- **Что важно понимать?**  
    Аргументы командной строки — это самый локальный слой конфигурации. Их лучше использовать для точечного override, а не как постоянное хранилище настроек.

- **Экосистема и альтернативы**

| Формат аргументов                        | Синтаксис            | Область применения                                     |
| ---------------------------------------- | -------------------- | ------------------------------------------------------ |
| `--Key=Value`                            | `--App:Timeout=300`  | ✅ Современный стандарт .NET                           |
| `--Key Value`                            | `--App:Timeout 300`  | ✅ Альтернативный синтаксис                            |
| `/Key=Value`                             | `/App:Timeout=300`   | ❌ Legacy Windows-совместимость                        |
| `ConfigurationBuilder.AddCommandLine()`  | Ручная регистрация   | ❌ Требуется при отказе от `CreateApplicationBuilder`  |

- **Типичные ошибки**
    - Используют CLI для постоянного хранения настроек вместо ENV/JSON.
    - Путают `--Key=Value` с `--Key Value` (пробел) и получают ошибки парсинга.
    - Не передают `args` в `CreateApplicationBuilder(args)`, из-за чего CLI-аргументы игнорируются.
    - Ожидают, что CLI-аргументы сохранятся между запусками.
    - Используют `/Key=Value` вместо `--Key=Value` на кроссплатформенных сценариях.

- **Production-практика**
    - Использовать CLI только для временного override при отладке и тестах.
    - Всегда передавать `args` в `CreateApplicationBuilder(args)`.
    - Документировать все доступные CLI-аргументы в README.
    - Для сценариев запуска создавать обёртки (shell-скрипты, .ps1), которые фиксируют аргументы.
    - Не хранить критичные данные (секреты, ключи) в CLI-аргументах (они видны в `ps aux`).

## Секция 4: Автоматизация проверки приоритетов

**Когда источники уже определены, важно не просто надеяться на правильный порядок, а зафиксировать его проверкой. Эта секция связывает конфигурационный пайплайн с повторяемым тестом.**

📚 **Теоретическая справка**

- **Что это?**  
    Инфраструктурный скрипт, который запускает приложение, подставляет разные источники конфигурации и проверяет итоговый вывод.

- **Зачем это?**  
    Он подтверждает, что порядок приоритетов работает стабильно, а значит конфигурационный пайплайн не сломается после изменений SDK, шаблона проекта или порядка регистрации провайдеров.

- **Как это реализовано?**  
    Скрипт временно задаёт переменные окружения, запускает `dotnet run`, собирает `stdout` и проверяет, какое значение реально прочитало приложение. Если итог не совпадает с ожидаемым приоритетом, скрипт возвращает ошибку.

- **Что важно понимать?**  
    Проверка должна быть повторяемой и независимой от ручных действий. Только так можно гарантировать, что приоритеты `CLI > ENV > JSON` сохраняются одинаково на разных машинах и в CI.

- **Экосистема и альтернативы**

| Метод валидации                                 | Уровень изоляции          | Сложность интеграции                               |
| ----------------------------------------------- | ------------------------- | -------------------------------------------------- |
| Оболочечный скрипт (PS/Bash)                    | Высокий (полный процесс)  | ✅ Низкая, нативная CI-интеграция                  |
| `xUnit` + `TestServer`/`WebApplicationFactory`  | Высокий (в памяти)        | ❌ Требует мок-хоста, избыточно для консоли        |
| `dotnet run` + ручной diff                      | Низкий                    | ❌ Не масштабируется, подвержен ошибкам            |
| Contract-тесты (Pact)                           | Средний                   | ❌ Сфокусировано на API-контрактах, не на конфиге  |

- **Типичные ошибки**
    - Пишут тесты, которые зависят от ручных действий (например, требуют установки ENV вручную).
    - Не сбрасывают ENV после теста, что ломает следующие прогоны.
    - Не проверяют все комбинации (JSON только, JSON+ENV, JSON+ENV+CLI).
    - Не возвращают код ошибки при*failе*, скрипт всегда возвращает 0.
    - Не включают скрипт в CI-пайплайн.

- **Production-практика**
    - Всегда сбрасывать ENV в конце скрипта (или использовать изолированные process).
    - Проверять все комбинации источников конфигурации.
    - Возвращать код `0` только при успешном прохождении всех тестов.
    - Включать скрипт в CI (GitHub Actions, GitLab CI, Azure Pipelines).
    - Логировать ожидания и фактические значения для отладки.

## Секция 5: Типизированный доступ через `IOptions<T>`

**После проверки источников логично перейти от "сырых" ключей к типизированному чтению настроек. Здесь конфигурация превращается в обычный объект, удобный для DI и валидации.**

> ⚠️ **Важная оговорка:** `IOptions<T>.Value` кэшируется при первом запросе, а не при старте хоста. Изменения в `appsettings.json` после первого резолва не подхватываются. Для hot-reload нужен `IOptionsMonitor<T>`.

📚 **Теоретическая справка**

- **Что это?**  
    `IOptions<T>` — это типизированный способ читать конфигурацию как объект, а не как набор строковых ключей.

- **Зачем это?**  
    Он уменьшает количество ошибок в ключах, делает код понятнее, поддерживает DI и позволяет валидировать настройки как обычный POCO-класс.

- **Как это реализовано?**  
    `Configure<T>` регистрирует фабрику в контейнере. При первом обращении к `IOptions<T>.Value` `ConfigurationBinder` сопоставляет поля и свойства класса с ключами конфигурации, приводит типы и кэширует результат. Повторные обращения используют уже созданный экземпляр.

- **Что важно понимать?**  
    `IOptions<T>` удобно там, где конфигурация стабильна на время работы приложения. Если значения должны обновляться без перезапуска, нужен `IOptionsMonitor<T>`.

- **Экосистема и альтернативы**

| Интерфейс доступа               | Стратегия обновления               | Область применения                                         |
| ------------------------------- | ---------------------------------- | ---------------------------------------------------------- |
| `IOptions<T>`                   | Единожды при старте (синглтон)     | ✅ Консольные приложения, статичный конфиг                 |
| `IOptionsSnapshot<T>`           | Перезагрузка на каждый скоуп       | ✅ Веб-запросы, где конфиг может меняться между запросами  |
| `IOptionsMonitor<T>`            | Реактивные обновления (`OnChange`) | ✅ Горячая перезагрузка без рестарта, фоновые сервисы      |
| `ConfigurationBinder.Get<T>()`  | Прямой вызов, без DI               | ❌ Ручное инстанцирование, не тестируется через моки       |

- **Типичные ошибки**
    - Используют `IOptions<T>` там, где нужен hot-reload (нужен `IOptionsMonitor<T>`).
    - Создают классы с nested-свойствами, которые не сопоставляются корректно.
    - Не валидируют настройки после биндинга (можгут быть null или неверные типы).
    - Ожидают, что `IOptions<T>` подхватит изменения в JSON без перезапуска.
    - Путают `IOptions<T>` с `IOptionsSnapshot<T>` и `IOptionsMonitor<T>`.

- **Production-практика**
    - Использовать `IOptions<T>` для консольных приложений со статичным конфигом.
    - Использовать `IOptionsMonitor<T>` для веб-приложений с hot-reload.
    - Добавлять валидацию через `IOptionsValidator` или ручной `Validate` при старте.
    - Создавать POCO-классы с `init`-свойствами для неизменяемости после биндинга.
    - Группировать настройки по секциям (например, "App", "Database", "Logging").

## Секция 6: Документация и сдача модуля

**Когда конфигурация уже собрана, переопределена и типизирована, остаётся зафиксировать контракт: что откуда берётся, как проверяется и как это передаётся дальше в команду.**

📚 **Теоретическая справка**

- **Что это?**  
    Формализация конфигурационных контрактов в технической документации и стандартизированный процесс интеграции изменений через изолированную ветку и код-ревью.

- **Зачем это?**  
    Это фиксирует ожидаемое поведение конфигурационного пайплайна, помогает новым разработчикам быстро понять правила приоритетов и сохраняет проверяемую историю изменений.

- **Как это реализовано?**  
    Документация описывает источники конфигурации, формат ключей, правила приоритетов и способы переопределения. Pull Request создаёт изолированный контекст ревью, запускает CI-проверки и сохраняет историю обсуждения, а коммиты с понятными префиксами помогают автоматизировать релизный процесс.

- **Что важно понимать?**  
    Для конфигурации особенно важно не просто "сделать, чтобы работало", а зафиксировать контракт: что является базой, что переопределением и что считается финальным значением.

- **Экосистема и альтернативы**

| Практика сдачи                 | Механизм                                               | Отличия от задачи                                     |
| ------------------------------ | ------------------------------------------------------ | ----------------------------------------------------- |
| PR с документацией контрактов  | Изолированное ревью, CI-валидация, исторический аудит  | ✅ Используется                                       |
| Direct commit в main           | Мгновенный деплой без ревью                            | ❌ Высокий риск поломки конфигурационного пайплайна   |
| Squash merge                   | Атомарное объединение коммитов                         | ❌ Теряется пошаговая история настройки провайдеров   |
| Release-ноты без PR            | Ручное описание изменений                              | ❌ Отсутствие автоматической привязки к коду и тестам |

- **Типичные ошибки**
    - Не документируют ENV-переменные и CLI-аргументы.
    - Открывают PR без описания приоритетов конфигурации.
    - Не включают скрипт автоматизации в PR.
    - Пишут коммиты без префиксов (`feat`, `fix`, `chore`).
    - Сливают без прогона CI-тестов.

- **Production-практика**
    - Всегда прикладывать README с примерами JSON, ENV, CLI.
    - Включать скрипт автоматизации в репозиторий и CI.
    - Использовать конвенцию коммитов (`feat(module3): ...`).
    - Описывать в PR: что изменилось, почему и как проверено.
    - Для серьёзных изменений конфигурации фиксировать краткий ADR (Architecture Decision Record).

## Критерий приёмки

- `appsettings.json` читается при запуске приложения.
- Переменная окружения `App__Timeout` (с `__`) переопределяет значение из JSON.
- Аргумент командной строки `--App:Timeout=300` имеет наивысший приоритет.
