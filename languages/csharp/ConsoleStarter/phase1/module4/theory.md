# Модуль 4 — .NET 10 Options & Runtime Reconfiguration

> Цель: Подписаться на изменения конфигурации в **.NET 10**, применять новые значения без рестарта и выстроить надёжный runtime reconfiguration-поток.

## Вступление

Этот модуль продолжает курс после модулей 1–3 и переводит конфигурацию из статического режима в **реактивный**. Если раньше приложение в .NET 10 просто читало значения при запуске, то теперь оно должно уметь отслеживать изменения файла конфигурации и безопасно применять их в рантайме.ingomoney+1  
Здесь важно не только читать новые значения, но и правильно валидировать конфиг, понимать разницу между `IOptions<T>`, `IOptionsMonitor<T>` и `IOptionsSnapshot<T>`, а также строить логику так, чтобы изменение файла не требовало перезапуска приложения.codewithmukesh+1

## Секция 1: `IOptionsMonitor<T>` как реактивный доступ

**В этой секции рассматривается `IOptionsMonitor<T>` в .NET 10 — основной механизм для получения актуальных значений конфигурации в долгоживущих сервисах и фоновых процессах.**

📚 **Теоретическая справка**

- **Что это?**  
   `IOptionsMonitor<T>` — интерфейс доступа к конфигурации, который возвращает актуальное значение через `CurrentValue` и поддерживает подписку на изменения.
- **Зачем это?**  
   Он позволяет сервисам в **.NET 10** подхватывать новые параметры без рестарта приложения и без потери состояния.
- **Как это реализовано?**  
   `IOptionsMonitor<T>` регистрируется в DI как singleton. При чтении `CurrentValue` он использует фабрику опций и текущий `IConfiguration`. Когда источник конфигурации меняется, провайдер сигнализирует о change token, и следующий доступ возвращает обновлённый экземпляр опций.
- **Что важно понимать?**  
   `IOptionsMonitor<T>` подходит для живых процессов, которые должны реагировать на изменения конфигурации во время работы. Это основной выбор для runtime reconfiguration в .NET 10.
- **Экосистема и альтернативы**

| Интерфейс доступа              | Стратегия обновления                               | Область применения                            |
| ------------------------------ | -------------------------------------------------- | --------------------------------------------- |
| `IOptionsMonitor<T>`           | Реактивная, актуальное значение при каждом запросе | ✅ Долгоживущие сервисы, hot-reload           |
| `IOptions<T>`                  | Один снимок при первом запросе                     | ❌ Статичная конфигурация, консольные утилиты |
| `IOptionsSnapshot<T>`          | Новый снимок на scope                              | ❌ Scoped-сценарии, веб-запросы               |
| Прямое чтение `IConfiguration` | Ручной парсинг без типизации                       | ❌ Legacy-код и быстрые прототипы             |

- **Типичные ошибки**
  - Используют `IOptions<T>` там, где нужен live update.
  - Ожидают, что конфиг обновится без `reloadOnChange: true`.
  - Читают опции один раз и кэшируют их навсегда.
  - Путают reactive-режим с scoped snapshot.
  - Не валидируют новые значения после обновления.
- **Production-практика**
  - Использовать `IOptionsMonitor<T>` для долгоживущих сервисов .NET 10.
  - Читать `CurrentValue` в местах, где важна актуальность.
  - Не хранить устаревшую копию опций дольше, чем нужно.
  - Подключать JSON-провайдер с `reloadOnChange: true`.
  - Документировать, какие параметры можно менять на лету.

## Секция 2: Подписка на `OnChange`

**Дальше нужно не только читать новые значения, но и реагировать на них. Для этого в .NET 10 используется `OnChange`, который позволяет запускать callback при обновлении конфигурации.**

📚 **Теоретическая справка**

- **Что это?**  
   `OnChange` — механизм подписки на изменения `IOptionsMonitor<T>`, который вызывает делегат при каждом успешном обновлении опций.
- **Зачем это?**  
   Он позволяет обновлять кэши, пересоздавать соединения, менять интервалы обработки или логировать изменение параметров без остановки приложения.
- **Как это реализовано?**  
   Подписка регистрируется в `IOptionsMonitor<T>`. При изменении конфигурационного источника JSON-провайдер с `reloadOnChange: true` фиксирует событие, пересчитывает конфигурацию и рассылает уведомление подписчикам.dev+1
- **Что важно понимать?**  
   `OnChange` нужен не для поллинга, а для событийной реакции. Это особенно удобно в NET 10, когда долгоживущий сервис должен перестроиться сразу после изменения файла.
- **Экосистема и альтернативы**

| Механизм отслеживания                       | Детерминизм                    | Накладные расходы         |
| ------------------------------------------- | ------------------------------ | ------------------------- |
| `OnChange`                                  | Высокий, событийный            | Низкие                    |
| Циклический опрос `CurrentValue`            | Средний, зависит от интервала  | Высокие                   |
| `GetReloadToken().RegisterChangeCallback()` | Высокий, низкоуровневый        | Средние                   |
| Внешние конфиг-системы                      | Высокий, зависит от провайдера | Зависит от инфраструктуры |

- **Типичные ошибки**
  - Подписываются на изменения, но не включают `reloadOnChange: true`.
  - Держат callback слишком тяжёлым и блокирующим.
  - Не отписываются от наблюдателя в долгоживущих объектах.
  - Путают изменение файла с немедленным обновлением всех зависимостей.
  - Ожидают, что `OnChange` сработает без изменения источника.
- **Production-практика**
  - Делать callback коротким и безопасным.
  - Логировать факт обновления и новое значение.
  - Использовать `OnChange` для реактивной части, а не для бизнес-логики целиком.
  - Подключать `reloadOnChange: true` только там, где hot-reload действительно нужен.
  - Тестировать изменение файла как отдельный сценарий.

## Секция 3: Валидация конфига при старте

**После того как конфигурация может обновляться, её нужно защитить от неверных значений. В .NET 10 это удобно делать через `ValidateDataAnnotations()` и `ValidateOnStart()`.**

📚 **Теоретическая справка**

- **Что это?**  
   Это механизм декларативной проверки опций по атрибутам и правилам ещё до запуска основного потока приложения.
- **Зачем это?**  
   Он реализует fail-fast: приложение сразу сообщает о плохой конфигурации, а не падает позже в бизнес-логике.
- **Как это реализовано?**  
   `ValidateDataAnnotations()` проверяет атрибуты вроде `[Required]` и `[Range]`. `ValidateOnStart()` заставляет `IOptions<T>` пройти валидацию при старте хоста, а не в момент первого случайного обращения.[codewithmukesh](https://codewithmukesh.com/blog/options-pattern-in-aspnet-core/)
- **Что важно понимать?**  
   В .NET 10 реактивная конфигурация без валидации опасна: новое значение может быть формально обновлено, но логически неприемлемо.
- **Экосистема и альтернативы**

| Стратегия валидации                               | Время выполнения      | Покрытие ошибок                |
| ------------------------------------------------- | --------------------- | ------------------------------ |
| `ValidateDataAnnotations()` + `ValidateOnStart()` | При старте            | ✅ Базовые правила и диапазоны |
| Ленивая валидация                                 | При первом обращении  | ❌ Ошибки проявляются поздно   |
| Кастомные `.Validate(...)`                        | При старте или лениво | ✅ Сложные бизнес-правила      |
| Ручная проверка в `Program.cs`                    | При старте            | ❌ Размытие ответственности    |

- **Типичные ошибки**
  - Не валидаируют обязательные поля.
  - Надеются, что обновление конфига всегда безопасно.
  - Откладывают проверку до runtime.
  - Смешивают валидацию и бизнес-обработку.
  - Не показывают понятную причину падения хоста.
- **Production-практика**
  - Включать валидацию на старте для всех критичных опций.
  - Использовать DataAnnotations для базовых контрактов.
  - Добавлять кастомную валидацию для кросс-полевых правил.
  - Проверять конфиг до запуска фоновых процессов.
  - Документировать, какие значения считаются допустимыми.

## Секция 4: Runtime reconfiguration в сервисах

**Когда монитор и валидация готовы, можно встроить динамические параметры в реальную рабочую логику. Здесь сервис в .NET 10 должен использовать новое значение без рестарта.**

📚 **Теоретическая справка**

- **Что это?**  
   Паттерн, при котором долгоживущий сервис читает актуальную конфигурацию во время работы и применяет её в следующем рабочем цикле.
- **Зачем это?**  
   Он позволяет менять интервалы, лимиты, флаги и режимы обработки без остановки приложения.
- **Как это реализовано?**  
   Сервис читает `_monitor.CurrentValue` на каждой итерации или обновляет локальный кэш через `OnChange`. В .NET 10 это безопасно и удобно для фоновых задач и worker-сценариев.[ingomoney](https://ingomoney.com/developer-blog/net-core-options-pattern-for-live-reload-settings/)
- **Что важно понимать?**  
   Runtime reconfiguration должна быть простой: изменение файла не должно ломать цикл обработки и не должно требовать ручного перезапуска.
- **Экосистема и альтернативы**

| Подход                               | Влияние на выполнение    | Сложность поддержки             |
| ------------------------------------ | ------------------------ | ------------------------------- |
| `CurrentValue` + `OnChange`          | Без перезапуска          | Низкая                          |
| Мягкий перезапуск через cancellation | Кратковременный downtime | Средняя                         |
| `static`-переменные                  | Без перезапуска          | Высокая, риск гонок             |
| Внешние feature flags                | Без перезапуска          | Высокая, требует инфраструктуры |

- **Типичные ошибки**
  - Кэшируют старое значение навсегда.
  - Читают опции только при старте.
  - Дублируют логику перезагрузки в каждом сервисе.
  - Путают hot-reload с перезапуском приложения.
  - Делают обновление конфигурации частью business flow.
- **Production-практика**
  - Читать актуальное значение только там, где оно нужно.
  - Делать reconfiguration максимально локальной.
  - Писать лог при смене параметров.
  - Проверять поведение цикла после обновления файла.
  - Ограничивать набор параметров, разрешённых к hot-reload.

## Секция 5: `IOptionsSnapshot<T>` как scoped-вариант

**Для сравнения важно показать и scoped-подход. В .NET 10 `IOptionsSnapshot<T>` полезен там, где нужно стабильное значение внутри одной операции, но допускается обновление между scope.**

📚 **Теоретическая справка**

- **Что это?**  
   `IOptionsSnapshot<T>` — scoped-вариант доступа к опциям, создающий новый снимок на каждый scope.
- **Зачем это?**  
   Он полезен, когда внутри одной операции значение должно оставаться неизменным, но между операциями может обновляться.
- **Как это реализовано?**  
   DI создаёт новый экземпляр при каждом scope, а `IOptionsFactory<T>` собирает текущую конфигурацию в POCO-объект.[code4it](https://www.code4it.dev/blog/ioptions-ioptionsmonitor-ioptionssnapshot/)
- **Что важно понимать?**  
   Это не replacement для `IOptionsMonitor<T>`. В .NET 10 snapshot нужен для scoped-стабильности, а monitor — для глобального live-update.
- **Экосистема и альтернативы**

| Стратегия              | Время жизни             | Консистентность                    |
| ---------------------- | ----------------------- | ---------------------------------- |
| `IOptionsSnapshot<T>`  | Scoped                  | ✅ Внутри одного scope             |
| `IOptionsMonitor<T>`   | Singleton               | ❌ Может измениться между вызовами |
| `IOptions<T>`          | Singleton-like snapshot | ❌ Не обновляется                  |
| Прямой параметр метода | Локальный               | ✅ Если передан явно               |

- **Типичные ошибки**
  - Используют snapshot в worker-сервисах вместо monitor.
  - Ожидают live-update внутри уже созданного scope.
  - Путают scoped-консистентность с глобальной актуальностью.
  - Держат scope слишком долго.
  - Используют snapshot там, где нужен runtime reconfiguration.
- **Production-практика**
  - Брать snapshot, когда важна неизменность на время операции.
  - Не использовать его как механизм hot-reload.
  - Чётко разделять scoped и singleton-сценарии.
  - Понимать жизненный цикл scope в приложении.
  - Использовать snapshot только там, где это действительно оправдано.

## Секция 6: Документация и сдача модуля

**Когда реактивная конфигурация уже работает, нужно зафиксировать правила обновления, валидации и применения изменений в документации и PR.**

📚 **Теоретическая справка**

- **Что это?**  
   Формализация контракта динамической конфигурации через документацию и процесс сдачи через Pull Request.
- **Зачем это?**  
   Это помогает команде понимать, какие параметры можно менять на лету и как система на это реагирует.
- **Как это реализовано?**  
   Изменения фиксируются в отдельной ветке, проходят PR-ревью и CI. Документация описывает hot-reload, `OnChange`, валидацию и ожидаемое поведение конфигурации.learn.microsoft+1
- **Что важно понимать?**  
   Для **.NET 10** документация — это часть архитектуры, а не постфактум-описание.
- **Экосистема и альтернативы**

| Практика сдачи                | Механизм                       | Отличия от задачи              |
| ----------------------------- | ------------------------------ | ------------------------------ |
| PR с документацией контрактов | Изолированное ревью, CI, аудит | ✅ Используется                |
| Direct commit в main          | Быстро, но без контроля        | ❌ Высокий риск регрессий      |
| Squash merge                  | Упрощает историю               | ❌ Теряет пошаговый контекст   |
| Release notes без PR          | Ручное описание                | ❌ Нет связки с кодом и тестом |

- **Типичные ошибки**
  - Оставляют hot-reload без документации.
  - Не объясняют, какие значения валидны.
  - Сливают изменения без проверки сценария обновления.
  - Не фиксируют контракт между конфигом и сервисом.
  - Пишут README как формальность.
- **Production-практика**
  - Описывать правила обновления конфига явно.
  - Документировать поддержку `OnChange` и `ValidateOnStart`.
  - Добавлять примеры изменения файла и реакции приложения.
  - Держать PR маленьким и проверяемым.
  - Сохранять инженерные решения в понятной форме.
