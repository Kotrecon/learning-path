## Секция 2: Структурированное логирование через `ILogger<T>`

### [x] Задача 6.2: Инжектить `ILogger<T>` в компоненты для наблюдаемости

- **Статус:** ✅ Completed
- **Дата выполнения:** 2026-05-27
- **Коммит:** `feat(module6.2): inject ILogger<PipelineWorker> with structured levels`
- **Заметки/Наблюдения:**
  - `ILogger<T>` инжектится через конструктор, тип `T` становится категорией лога
  - Уровни: `Debug` (детали), `Information` (нормальный ход), `Warning` (не критично), `Error` (сбой)
  - Шаблонизированные сообщения (`"Processing {Timeout}ms"`) позволяют агрегаторам индексировать параметры отдельно от текста
  - 💡 Инсайт: логирование — это не «написать в консоль», а «сделать поведение приложения наблюдаемым для человека и машины».

## Секция 3: Базовая обработка необработанных исключений

### [x] Задача 6.3: Реализовать try/catch в Main для логирования критических ошибок

- **Статус:** ✅ Completed
- **Дата выполнения:** 2026-05-27
- **Коммит:** `feat(module6.3): add global error handler for startup failures`
- **Заметки/Наблюдения:**
  - `BackgroundService` исключения обрабатываются хостом внутри (`StopHost` по умолчанию) → `RunAsync()` возвращается штатно, exit code `0`
  - Глобальный `try/catch` в `Main` ловит **только** ошибки инициализации (конфиг, DI, билд), не ошибки в `ExecuteAsync`
  - Для строгого `ExitCode = 1` при падении воркера нужен паттерн с `IErrorTracker` или валидация до `RunAsync()`
  - 💡 Инсайт: `BackgroundService` — для «терпимых» фоновых задач. Критичная логика старта должна валидироваться до `RunAsync()`.

## Секция 5: Подготовка к деплою через dotnet publish

### [x] Задача 6.5: Настроить публикацию self-contained билда для продакшена

- **Статус:** ✅ Completed
- **Дата выполнения:** 2026-05-27
- **Коммит:** `feat(module6.5): add self-contained publish config & production launch note`
- **Заметки/Наблюдения:**
  - `--self-contained true` включает Runtime, запуск возможен без .NET SDK на целевой машине
