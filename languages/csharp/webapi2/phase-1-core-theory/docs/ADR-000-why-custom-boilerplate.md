# ADR-000: Почему для микросервиса выбираем custom boilerplate с DDD

## Статус

Принято

## Контекст

Нужно создать шаблон (boilerplate) микросервиса. Существует три варианта:

1. **Custom boilerplate** — создавать с нуля
2. **Готовый шаблон** — взять существующий (например, Jason Taylor Clean Architecture Template)
3. **Гибрид** — взять готовый, адаптировать под себя

## Решение

Выбираем **custom boilerplate с DDD структурой**.

## Обоснование

### Почему custom

- **Полный контроль** — понимаем каждую строку кода
- **Нет лишних зависимостей** — только то, что нужно
- **Понимание архитектуры** — знаем как всё работает изнутри
- **Нет привязки к автору шаблона** — свой проект, свои решения

### Почему DDD (Domain-Driven Design)

- **Структура отражает бизнес-домен** — код понятен бизнесу
- **Чёткие границы** — Aggregates, Entities, Value Objects
- **Domain Layer — самый внутренний слой** — не зависит ни от кого
- **Shared.Config — независимый проект** — общая конфигурация + конфигураторы

## Альтернативы (что отвергли)

| Альтернатива                  | Почему отвергли                                                                  |
| ----------------------------- | -------------------------------------------------------------------------------- |
| Готовый шаблон (Jason Taylor) | Избыточность для простых проектов, сложность кастомизации, зависимость от автора |
| Clean Architecture            | Усложнение для boilerplate, DDD проще и достаточен                               |

## Последствия

### Положительные

- Полный контроль над архитектурой
- Понимание каждой строки
- Нет лишних зависимостей
- Boilerplate можно клонировать и наполнять бизнес-логикой

### Отрицательные

- Нужно время на создание
- Риск ошибок при проектировании

## Принятые правила

1. **Secrets никогда не в коде** — User Secrets (dev), Environment Variables (prod), Vault (enterprise)
2. **Middleware порядок критичен** — Exception Handler → HTTPS → CORS → Rate Limiting → Auth → Routing
3. **DI lifetime rules** — не инжектировать scoped в singleton
4. **Fail-fast** — быстрое падение при ошибках конфигурации
5. **Program.cs минимум** — только вызовы конфигураторов (≤20 строк)
6. **Вся конфигурация в Shared.Config** — через конфигураторы
